## Context

Cerebro is a complex Python application with multiple concerns:

**Python Dependency Management:**
- 30+ Python packages (FastAPI, Textual, sentence-transformers, typer, etc.)
- Version constraints and lock files
- Development vs production dependencies
- Poetry provides excellent Python-specific tooling

**System-Level Reproducibility:**
- System dependencies (git, nodejs, npm for dashboard)
- Python interpreter version (3.11+)
- Native libraries (SSL, database drivers)
- Development tools (ruff, pytest)
- Need for identical environments across machines

**The Problem:** Poetry alone doesn't guarantee reproducible system-level environments. `poetry.lock` ensures Python package versions, but not:
- Python interpreter version
- System libraries (OpenSSL versions differ)
- Native dependencies
- Development tools
- Environment variables

Traditional approaches (pip+requirements.txt, conda, docker) have limitations:
- **pip/venv:** No system dependencies, no reproducibility
- **Conda:** Slower, larger, Python-ecosystem focused
- **Docker:** Heavy, not suitable for development workflows
- **pyenv + Poetry:** Still doesn't pin system deps

## Decision

**Adopt dual-stack approach: Poetry for Python dependencies + Nix for system-level reproducibility.**

**Division of responsibilities:**
- **Poetry:** Python package management, lock file, virtual environments
- **Nix:** System dependencies, Python interpreter, dev tools, reproducible shell

**Implementation:**
```nix
# flake.nix
{
  inputs = {
    nixpkgs.url = "github:NixOS/nixpkgs/nixos-unstable";
    poetry2nix.url = "github:nix-community/poetry2nix";
  };

  outputs = { self, nixpkgs, poetry2nix }: {
    devShells.x86_64-linux.default = pkgs.mkShell {
      buildInputs = [
        python311      # Pinned interpreter
        poetry         # Python deps
        nodejs         # For dashboard
        git            # Version control
        ruff           # Linting
      ];

      shellHook = ''
        export POETRY_VIRTUALENVS_IN_PROJECT=true
        poetry install
      '';
    };
  };
}
```

Workflow:
```bash
nix develop          # Enter reproducible env
poetry install       # Install Python deps
cerebro              # Run application
```

## Rationale

### Drivers

- **Reproducibility:** Identical environments across dev/CI/prod
- **System Dependencies:** Need nodejs, git, system libraries
- **Python Best Practices:** Poetry is Python community standard
- **Developer Experience:** Fast iteration, familiar tools
- **NixOS Integration:** Team uses NixOS, leverage ecosystem
- **Future-Proofing:** Nix enables advanced deployment (containers, VMs, etc.)

### Alternatives Considered

#### Option A: Poetry Only
- **Pros:** Simple, one tool, Python-native
- **Cons:** No system dependencies, no reproducibility guarantee, Python version drift
- **Why rejected:** Can't pin system libraries, interpreter version not enforced

#### Option B: Nix Only (nixpkgs Python)
- **Pros:** Complete reproducibility, one tool
- **Cons:** Slower iteration (rebuild on dep change), less flexible than Poetry, learning curve
- **Why rejected:** Poor developer experience for Python development, slow dependency updates

#### Option C: Docker + Poetry
- **Pros:** Industry standard, reproducible, CI-friendly
- **Cons:** Heavy (GB images), slow startup, poor local dev experience, WSL complexity
- **Why rejected:** Overhead too high for development workflow

#### Option D: Conda + Nix
- **Pros:** Conda handles Python + some system deps
- **Cons:** Slow solver, large packages, overlapping with Nix, community fragmentation
- **Why rejected:** Redundant with Nix, slower than Poetry

#### Option E: Pipenv + Nix
- **Pros:** Similar to Poetry
- **Cons:** Pipenv less maintained, slower resolver, smaller community
- **Why rejected:** Poetry is more actively developed and faster

### Trade-offs

- **Accepted:** Learning curve for Nix vs single-tool simplicity
- **Accepted:** Two tools to maintain vs all-in-one solution
- **Accepted:** Nix complexity vs Docker complexity (chose Nix for NixOS team)
- **Benefit:** Best-of-breed for each layer (Poetry for Python, Nix for system)

## Consequences

### Positive

- **Guaranteed Reproducibility:** Bit-for-bit identical environments
- **Fast Python Iteration:** Poetry's fast resolver + caching
- **System Dependencies Solved:** nodejs, git, libraries all pinned
- **Familiar Tools:** Poetry is Python standard, no vendor lock-in for Python code
- **NixOS Synergy:** Team already uses NixOS, leverages existing knowledge
- **Future Options:** Nix enables containers, VMs, cloud deployment patterns
- **Lock Files:** Both `poetry.lock` (Python) and `flake.lock` (system) tracked in git

### Negative

- **Two Tools:** Must understand both Poetry and Nix
- **Learning Curve:** Nix has steep learning curve
- **Overhead:** Two lock files, two update processes
- **Complexity:** More moving parts than single-tool solution
- **CI Setup:** Must install Nix in CI (though cachix helps)
- **Team Onboarding:** New developers must learn Nix

### Risks

| Risk | Probability | Impact | Mitigation |
|------|-------------|--------|------------|
| Nix learning curve blocks development | Medium | High | Comprehensive docs, Nix templates, team training sessions |
| Poetry-Nix integration issues | Low | Medium | poetry2nix is mature, active community support |
| Lock file conflicts (poetry.lock vs flake.lock) | Low | Low | Clear workflows: Poetry for Python, Nix for system |
| Nix evaluation slowness | Low | Medium | Use nix-direnv for caching, binary cache for builds |
| Dependency update complexity | Medium | Low | Document update process, automated dependency PRs |

## Implementation

### Structure

```
cerebro/
├── flake.nix              # Nix system dependencies
├── flake.lock             # Nix lock file
├── pyproject.toml         # Poetry Python dependencies
├── poetry.lock            # Poetry lock file
└── .envrc                 # direnv auto-load (optional)
```

### Workflow

**Initial setup:**
```bash
git clone cerebro
cd cerebro
nix develop              # Enter Nix shell (downloads system deps)
# Poetry automatically installs Python deps via shellHook
```

**Daily development:**
```bash
nix develop              # Or use direnv
poetry run cerebro       # Run app
poetry add requests      # Add Python dep
poetry lock              # Update lock
```

**Update dependencies:**
```bash
# Python deps
poetry update
poetry lock

# System deps
nix flake update
```

**CI/CD:**
```yaml
- uses: cachix/install-nix-action@v22
- run: nix develop -c poetry install
- run: nix develop -c pytest
```
