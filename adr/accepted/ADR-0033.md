## Context

O adr-ledger tem 8 modulos `.chain/` implementando blockchain privado, mas nao rastreia suas proprias dependencias de forma estruturada. O Nix ja pina tudo via `flake.lock`, mas falta a camada de **governanca**: qual ADR justificou cada dependencia, quem aprovou, quando mudou.

Industry standards (CycloneDX, SPDX) exigem SBOMs para supply chain security, mas nenhuma ferramenta existente mapeia dependencias para decisoes arquiteturais.

## Decision

Implementar um SBOM decision-aware que enriquece (nao duplica) o que o Nix ja oferece:

1. **`sbom_manager.py`** — Modulo Python que escaneia `flake.lock`, `flake.nix` e `.chain/*.py` para gerar inventario completo
2. **Bootstrap ADR Map** — Dict mapeando dependencias existentes para ADRs que as justificam
3. **CycloneDX 1.6 Export** — Usando `properties[]` para ADR references (mecanismo padrao de extensao)
4. **Drift Detection** — Re-scan + comparacao com SBOM atual para detectar mudancas nao declaradas
5. **Snapshot Integration** — `sbom_hash` incluido nos snapshots criptograficos
6. **Governance Contract** — `check_supply_chain_compliance()` verifica que todas deps tem ADR reference

Escopo: CycloneDX + formato interno, nivel governance-aware, Nix como fonte de verdade.

## Rationale

### Drivers

- Nix pina deps mas nao tem camada de governance
- Precisamos saber QUAL decisao justificou CADA dependencia
- CycloneDX e o padrao mais adotado para SBOM
- Drift detection e essencial para supply chain integrity

### Alternatives Considered

#### Option A: Pure CycloneDX (sem ADR enrichment)
- **Pros:** Standard tooling, wide ecosystem support
- **Cons:** No governance context, misses decision-aware angle
- **Why rejected:** Loses the core value proposition

#### Option B: External tool (syft, trivy)
- **Pros:** Mature tooling, container scanning
- **Cons:** Cannot map deps to ADRs, extra dependency, not Nix-aware
- **Why rejected:** Cannot provide ADR references

### Trade-offs

- Custom SBOM manager adds ~400 LOC but provides unique governance integration
- Bootstrap ADR map is manual but covers all existing dependencies

## Consequences

### Positive

- Every dependency traceable to an ADR justification
- CycloneDX 1.6 export for standard tooling interoperability
- Drift detection catches undeclared dependency changes
- Snapshot integration links SBOM state to chain state

### Negative

- Additional module to maintain
- Bootstrap map requires manual updates for pre-existing deps

### Risks

| Risk | Probability | Impact | Mitigation |
|------|-------------|--------|------------|
| Bootstrap ADR map becomes stale | low | low | Frontmatter supply_chain auto-populates for new ADRs |

## Implementation

### Components

- `.chain/sbom_manager.py` — Core module (~400 LOC)
- `.schema/adr.schema.json` — `supply_chain` section added
- `.governance/governance.yaml` — `supply_chain` config block
- `scripts/adr` — `cmd_sbom()` CLI command
- `.chain/snapshot_manager.py` — `sbom_hash` field
- `.chain/governance_engine.py` — `check_supply_chain_compliance()`
- `flake.nix` — `sbom-valid` check

### CLI Usage

```bash
adr sbom generate [--signer X]   # Generate SBOM
adr sbom verify                   # Detect drift
adr sbom diff <old> [new]        # Compare SBOMs
adr sbom export [-o file]        # CycloneDX export
adr sbom show                    # Show current SBOM
adr sbom list-components [--type T]  # List components
```
