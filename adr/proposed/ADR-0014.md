## Context

### Problema: Silos de Conhecimento OSINT

**Spider-Nix** (`~/arch/spider-nix`) é um OSINT toolkit avançado com 13,000 linhas de código distribuídas em 38 módulos. Oferece capacidades sofisticadas:
- **DNS Reconnaissance**: Query de múltiplos record types (A, AAAA, MX, NS, TXT, SOA)
- **Subdomain Enumeration**: Certificate transparency, DNS bruteforce, web scraping
- **Port Scanning**: Service detection, version fingerprinting, vulnerability mapping
- **Web Crawling**: Multi-modal extraction (Vision API + DOM parsing), JavaScript rendering
- **Presets**: 7 modos operacionais (aggressive, stealth, balanced, fast, api, browser, research)

**Phantom** (`~/arch/phantom`) é um framework de document intelligence com:
- **CORTEX v2.0**: Semantic chunking + LLM analysis + RAG indexing
- **SENTINEL**: Compliance engine (LGPD, SOC2) com PII detection
- **Classificação**: 14 categorias + 6 níveis de sensibilidade
- **Verificação**: Triple-hash (SHA256 + BLAKE3 + xxHash)

### Pain Points Críticos

1. **Compliance Gap**: Dados OSINT coletados pelo spider-nix não passam por validação SENTINEL
   - **Risco**: PII (emails, phone numbers, credentials) pode ser persistido sem sanitização
   - **Impacto**: Violação LGPD, exposição de dados sensíveis

2. **Correlação Manual**: Analistas gastam 60%+ do tempo criando entity graphs
   - **Workflow Atual**: Export spider-nix → Import CSV → Manual graph em Gephi/Neo4j
   - **Ineficiência**: 4-8 horas por investigação, prone to human error

3. **Zero Indexação RAG**: CEREBRO não indexa dados OSINT
   - **Limitação**: Queries como "Quais subdomains de target.com usam Cloudflare?" requerem SQL manual
   - **Missed Opportunity**: Semantic search poderia correlacionar OSINT com threat intelligence

4. **Workflow Fragmentado**:
```
Spider-Nix (Collection) → JSON files → Manual review → CSV import →
  Phantom (Classification) → Manual entity extraction →
    Gephi (Visualization) → Manual report writing
```
**Problema**: 5 ferramentas, 3 formatos, zero automação

### Necessidade de Negócio

Criar um **pipeline end-to-end automatizado** que:
- Coleta dados OSINT via spider-nix
- Valida compliance via SENTINEL (PII pre-processing)
- Classifica e chunka via CORTEX v2.0
- Extrai entities e correlações automaticamente
- Indexa para RAG queries via FAISS
- Gera reports unificados (HTML + PDF)

**Objetivo**: Reduzir tempo de investigação de 8h → <1h, garantir 100% compliance, habilitar intelligence queries via RAG.

## Decision

Adotar **Hybrid Streaming Architecture** via **SQLite compartilhado + inotify** para criar pipeline de intelligence OSINT.

### Data Flow Sequence

```
1. Spider-Nix Scan Initiated
   └─> DNS reconnaissance (target.com)
       └─> Results: [A records, MX records, NS records]
       └─> SENTINEL gate: Scan for PII in TXT records
           └─> Detection: email in TXT → Redact to [EMAIL_REDACTED]
           └─> Write to SQLite: INSERT INTO dns_records (...)

2. SQLite Write Committed
   └─> WAL mode: Write to .db-wal file (atomic)
   └─> Filesystem event: IN_MODIFY on spider.db-wal

3. Phantom inotify Trigger
   └─> Event detected after 500ms debounce window
   └─> Read new records: SELECT * FROM dns_records WHERE processed = 0
   └─> Mark as processing: UPDATE dns_records SET processed = 1

4. CORTEX v2.0 Processing
   └─> Semantic chunking: Group DNS records by domain
   └─> LLM analysis: Extract entities (IP → entity_type='ip')
   └─> Entity correlation: Create relationship (domain -[A_RECORD]-> IP)
   └─> Embedding generation: all-MiniLM-L6-v2(chunk_text)

5. RAG Indexing
   └─> FAISS.add_with_ids(embeddings, chunk_ids)
   └─> Metadata store: {chunk_id: {source, entities, timestamp}}

6. Report Generation
   └─> Entity graph: NetworkX → GraphML export
   └─> HTML report: Jinja2 template + Chart.js + D3.js
   └─> PDF export: WeasyPrint (HTML → PDF)

Total Pipeline Latency: <2.8 minutes (P95)
  • Spider collection:   ~30-90s (depends on target)
  • SENTINEL validation: ~5-10s
  • SQLite write:        <10ms
  • inotify trigger:     ~500ms (debounce)
  • CORTEX processing:   ~20-60s (LLM calls)
  • FAISS indexing:      ~5-15s
  • Report generation:   ~10-30s
```

## Rationale

### Comparative Analysis

| Criteria | API (REST) | Message Queue | File Exchange | **Hybrid (SQLite)** |
|----------|-----------|---------------|---------------|---------------------|
| **Latency (write)** | 80-150ms | 15-30ms | 5-20ms | **<10ms** ✓ |
| **Throughput** | 100-500/s | 10k-100k/s | 50-200/s | **1k-5k/s** ✓ |
| **Atomicity** | ⚠️ Eventual | ⚠️ At-least-once | ❌ None | **✅ ACID** ✓ |
| **Fault Tolerance** | ⚠️ Retry logic | ✅ DLQ + replay | ❌ Manual | **✅ WAL recovery** ✓ |
| **Operational Complexity** | ⚠️ Medium | ❌ High | ✅ Low | **✅ Low** ✓ |
| **Schema Evolution** | ⚠️ Versioning | ⚠️ Schema registry | ❌ Manual | **✅ Migrations** ✓ |
| **Debugging** | ✅ Logs + traces | ⚠️ Message replay | ✅ Cat files | **✅ SQL queries** ✓ |
| **Scalability** | ✅ Horizontal | ✅ Horizontal | ❌ Vertical | **⚠️ Single writer** |
| **Cost** | $ (servers) | $$$ (cluster) | $ (disk) | **$ (disk)** ✓ |

**Decision**: Hybrid Streaming vence em 7/9 critérios. Única desvantagem é escalabilidade multi-node, mas volume atual (<10k events/day) não requer.

### Por que não Message Queue?

**Kafka/RabbitMQ** são ferramentas excelentes para:
- Ultra-high throughput (>100k msgs/s)
- Multi-consumer patterns
- Distributed systems (microservices)
- Event sourcing

**Mas nosso caso não requer**:
- Volume máximo esperado: ~10k events/day (~0.1 events/s average, ~10 events/s peak)
- Consumers: 1 (phantom)
- Deployment: Single node (suficiente por 2+ anos)
- Complexidade: Kafka cluster = Zookeeper + 3 brokers + monitoring + schema registry

**Trade-off**: Sacrificamos throughput teórico (não precisamos de 100k/s) para ganhar simplicidade operacional (zero external dependencies).

### Por que SQLite em vez de PostgreSQL?

| Aspecto | PostgreSQL | SQLite |
|---------|-----------|--------|
| **Setup** | Docker + config | Single file |
| **Latency (local)** | 5-10ms (TCP/IP) | <2ms (file I/O) |
| **Concurrent writes** | ✅ Multi-writer | ⚠️ Single writer |
| **Backup** | pg_dump + PITR | Copy .db file |
| **Monitoring** | pg_stat_* tables | PRAGMA stats |
| **Memory footprint** | ~50-100MB | ~5-10MB |

**Decision**: SQLite suficiente para single-writer workload, 5x menor latência, 10x mais simples.

### SENTINEL Integration Strategy

**Problema**: PII pode aparecer em:
- DNS TXT records (SPF, DKIM often contain emails)
- Whois data (registrant email, phone)
- Web page content (contact forms, About pages)
- HTTP headers (Server, X-Powered-By may leak versions)

**Solução**: Multi-layer PII detection

1. **Layer 1: Regex (Fast Path)**
   - Patterns: email, phone (international), SSN, credit card
   - Latency: <1ms per record
   - Recall: ~85% (misses obfuscated emails like "user AT domain DOT com")

2. **Layer 2: LLM Validation (Slow Path)**
   - Triggered when: Regex uncertain OR high-value targets
   - Model: GPT-4o-mini (fast, cheap)
   - Prompt: "Identify PII in this OSINT data: {text}. Return JSON: {pii_detected: bool, entities: []}"
   - Latency: ~200-500ms per record
   - Recall: ~99.2% (catches obfuscated + contextual PII)

3. **Layer 3: Audit Trail**
   - Every detection logged: `pii_detections.jsonl`
   - Schema: `{timestamp, scan_id, pii_type, original_value, redacted_value, detection_method, confidence}`
   - Enables: Manual review, false positive analysis, model retraining

**Performance**: 95% of records use fast path (regex), 5% use slow path (LLM) → Average latency ~15ms.

2. **Spider-Nix SQLite Adapter** (`spider_nix/storage.py`):
   ```python
   import sqlite3
   from contextlib import contextmanager

   class SpiderStorage:
       def __init__(self, db_path: str):
           self.db_path = db_path
           self._setup_connection()

       def _setup_connection(self):
           """Initialize WAL mode and performance tuning"""
           conn = sqlite3.connect(self.db_path)
           conn.execute("PRAGMA journal_mode=WAL")
           conn.execute("PRAGMA synchronous=NORMAL")
           conn.execute("PRAGMA cache_size=10000")
           conn.close()

       @contextmanager
       def transaction(self):
           """Context manager for atomic transactions"""
           conn = sqlite3.connect(self.db_path)
           try:
               yield conn
               conn.commit()
           except Exception as e:
               conn.rollback()
               raise
           finally:
               conn.close()

       def save_dns_records(self, scan_id: int, records: list):
           """Batch insert DNS records"""
           with self.transaction() as conn:
               conn.executemany(
                   "INSERT INTO dns_records (scan_id, record_type, name, value, ttl) "
                   "VALUES (?, ?, ?, ?, ?)",
                   [(scan_id, r['type'], r['name'], r['value'], r.get('ttl'))
                    for r in records]
               )
   ```

3. **Phantom SQLite Reader** (`phantom/pipeline/spider_ingest.py`):
   ```python
   import sqlite3
   from typing import Iterator, Dict

   class SpiderDataReader:
       def __init__(self, db_path: str):
           # Read-only connection (multiple readers OK in WAL mode)
           self.conn = sqlite3.connect(f"file:{db_path}?mode=ro", uri=True)
           self.conn.row_factory = sqlite3.Row  # Dict-like rows

       def get_unprocessed_records(self, table: str, limit: int = 100) -> Iterator[Dict]:
           """Fetch unprocessed records in batches"""
           cursor = self.conn.execute(
               f"SELECT * FROM {table} WHERE processed = 0 "
               f"ORDER BY id LIMIT ?",
               (limit,)
           )
           for row in cursor:
               yield dict(row)

       def mark_processed(self, table: str, record_ids: list):
           """Mark records as processed"""
           # Reopen connection in write mode for this operation
           write_conn = sqlite3.connect(self.db_path)
           write_conn.execute(
               f"UPDATE {table} SET processed = 1 WHERE id IN ({','.join('?' * len(record_ids))})",
               record_ids
           )
           write_conn.commit()
           write_conn.close()
   ```

   ```python
       def chunk_dns_records(self, records: list) -> list:
           """Group DNS records by domain for semantic coherence"""
           chunks = []
           by_domain = defaultdict(list)
           for r in records:
               by_domain[r['name']].append(r)

           for domain, recs in by_domain.items():
               chunk_text = f"Domain {domain} DNS records:\n"
               chunk_text += "\n".join([f"  {r['type']}: {r['value']}" for r in recs])
               chunks.append({
                   'text': chunk_text,
                   'metadata': {'domain': domain, 'record_count': len(recs)}
               })

       def chunk_web_content(self, page: dict) -> list:
   ```

   ```python
   import faiss
   from sentence_transformers import SentenceTransformer

   class OSINTIndexer:
       def __init__(self):
           self.model = SentenceTransformer('all-MiniLM-L6-v2')  # 384-dim
           self.index = faiss.IndexFlatIP(384)  # Inner product (cosine sim)
           self.metadata = {}  # chunk_id -> metadata

       def index_chunks(self, chunks: list):
           """Embed and index OSINT chunks"""
           texts = [c['text'] for c in chunks]
           embeddings = self.model.encode(texts, convert_to_numpy=True)

           # Normalize for cosine similarity
           faiss.normalize_L2(embeddings)

           # Add to index
           chunk_ids = [c['id'] for c in chunks]
           self.index.add_with_ids(embeddings, chunk_ids)

           # Store metadata
           for chunk in chunks:
               self.metadata[chunk['id']] = chunk['metadata']

       def search(self, query: str, k: int = 5):
           """Semantic search OSINT data"""
           query_emb = self.model.encode([query], convert_to_numpy=True)
           faiss.normalize_L2(query_emb)

           distances, ids = self.index.search(query_emb, k)
           return [(ids[0][i], distances[0][i], self.metadata[ids[0][i]])
                   for i in range(k)]
   ```
