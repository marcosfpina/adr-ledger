## Context

Cerebro is evolving into a multi-interface platform with TUI, CLI, and GUI modes. The problem: heavyweight dependencies like Textual (TUI framework) and ML models add significant startup overhead:

- Textual: ~200ms import time + 500KB package
- Sentence transformers: ~2-3s import + 500MB models
- FastAPI/Uvicorn: ~100ms for GUI mode

When users invoke simple CLI commands like `cerebro knowledge analyze .`, they shouldn't pay the penalty of loading TUI widgets or ML models. The CLI should remain fast and lightweight for scripting use cases.

**Baseline measurements:**
- Pure CLI startup: ~50ms
- With eager TUI import: ~250ms (+400%)
- With eager ML models: ~3.2s (+6300%)

## Decision

**Implement lazy import strategy for heavyweight dependencies: defer imports until actual use.**

Pattern:
```python
# BEFORE (eager)
from phantom.tui.app import CerebroApp  # Always loaded

# AFTER (lazy)
def launch_tui():
    from phantom.tui.app import CerebroApp  # Only on TUI invocation
    app = CerebroApp()
    app.run()
```

Apply to:
1. Textual framework (TUI mode only)
2. ML models (RAG/intelligence commands only)
3. FastAPI/Uvicorn (GUI mode only)
4. Heavy data processing libraries (pandas, when used)

## Rationale

### Drivers

- CLI commands must remain fast for scripting (<100ms startup)
- TUI dependencies irrelevant for 90% of CLI usage
- ML models only needed for intelligence/RAG commands
- Users shouldn't wait for unused subsystems
- Maintain single unified codebase (no package splitting)

### Alternatives Considered

#### Option A: Eager Loading (current)
- **Pros:** Simple imports, no dynamic loading complexity
- **Cons:** 3+ second CLI startup with all deps, terrible scripting experience
- **Why rejected:** Makes CLI unusable for automation/scripts

#### Option B: Split Packages (cerebro-core, cerebro-tui, cerebro-ml)
- **Pros:** Clean dependency separation, pip-installable subsets
- **Cons:** Complex packaging, dependency conflicts, harder maintenance, user confusion
- **Why rejected:** Over-engineering for small project, maintenance burden

#### Option C: Separate Entry Points (cerebro-cli, cerebro-tui binaries)
- **Pros:** Clear separation, independent optimization
- **Cons:** UX fragmentation, multiple commands to learn, code duplication
- **Why rejected:** Violates unified platform goal

#### Option D: Plugin System
- **Pros:** Ultimate extensibility, clean boundaries
- **Cons:** Complex architecture, runtime discovery overhead, overkill
- **Why rejected:** Too complex for current scale

### Trade-offs

- **Accepted:** Slight code complexity (function-scoped imports) vs startup performance
- **Accepted:** Import errors delayed to runtime vs upfront (mitigated by tests)
- **Accepted:** Less obvious dependencies vs faster CLI
- **Benefit:** Keep unified codebase with optimized per-mode performance

## Consequences

### Positive

- **Fast CLI:** Maintains <100ms startup for simple commands
- **Efficient Resource Use:** Only load what's needed for current operation
- **Unified Codebase:** Single package with smart loading
- **Better UX:** Scripts/automation remain fast, interactive TUI loads when needed
- **Flexible Deployment:** Can run on systems without TUI/GUI deps for CLI-only usage

### Negative

- **Import Errors Delayed:** Dependency issues discovered at runtime, not startup
- **Code Duplication:** Import statements scattered in multiple functions
- **Debugging Complexity:** Stack traces may show import inside function calls
- **Testing Burden:** Must test each code path to ensure imports work
- **IDE Experience:** Auto-completion may not work as well for lazy imports

### Risks

| Risk | Probability | Impact | Mitigation |
|------|-------------|--------|------------|
| Missing dependency at runtime | Medium | High | Comprehensive integration tests for each command path |
| Import statement duplication | Medium | Low | Document pattern, use ruff to detect issues |
| Circular import issues | Low | Medium | Careful module structure, import at function scope |
| Performance regression from repeated imports | Low | Low | Python caches imports (sys.modules), negligible overhead |

## Implementation

### Pattern

**Standard lazy import pattern:**
```python
def launch_tui():
    """Launch TUI - heavy deps only loaded here."""
    try:
        from phantom.tui.app import CerebroApp
    except ImportError as e:
        raise RuntimeError(
            "TUI dependencies not available. "
            "Install with: poetry install --extras tui"
        ) from e

    app = CerebroApp()
    app.run()
```

**Applied to:**

1. **TUI Mode** (`src/phantom/launcher.py`):
   ```python
   if mode == "tui":
       from phantom.tui.app import CerebroApp  # Lazy
       CerebroApp().run()
   ```

2. **ML Models** (`src/phantom/knowledge/engine.py`):
   ```python
   def load_model(self):
       from sentence_transformers import SentenceTransformer  # Lazy
       self.model = SentenceTransformer(self.model_name)
   ```

3. **GUI Mode** (`src/phantom/launcher.py`):
   ```python
   if mode == "gui":
       import uvicorn  # Lazy
       uvicorn.run("phantom.api.server:app")
   ```
