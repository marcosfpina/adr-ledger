## Context

SPECTRE runs multiple services in Kubernetes: spectre-proxy (Rust/axum), neutron (Python/FastAPI), and NATS (message bus). As inter-service communication grows, we need:

1. **Mutual TLS (mTLS)** between pods without modifying application code
2. **Observability** at the network layer (latency, throughput, success rate per route)
3. **Traffic management** for canary deployments and gradual rollouts

The proxy already implements circuit breaker and rate limiting in Rust code. The mesh complements these by providing network-layer security and observability.

## Decision

**Adopt Linkerd** as the service mesh for SPECTRE's Kubernetes deployments.

Key configuration:
- **iptables mode**: `nft` (required for kind clusters, compatible with modern kernels)
- **NATS skip**: `config.linkerd.io/skip-outbound-ports: 4222` (NATS uses a custom TCP protocol, not HTTP)
- **Injection**: Annotation-based (`linkerd.io/inject: enabled`) per deployment, not namespace-wide
- **Coexistence**: Application-level circuit breaker (Rust) + mesh-level mTLS/retries

## Rationale

### Drivers

- **Zero-trust networking**: mTLS between all pods without application code changes
- **Operational simplicity**: Linkerd's Rust-based proxy has minimal memory footprint (~15MB vs Istio's ~50MB+)
- **Observability**: Golden metrics (p50/p95/p99 latency, RPS, success rate) per service pair

### Alternatives Considered

#### Option A: Istio
- **Pros:** Most feature-rich (VirtualService, AuthPolicy, Wasm extensions)
- **Cons:** Heavy sidecar (Envoy, 50MB+), complex CRDs, steep learning curve
- **Why rejected:** Overkill for current service count (2-3 services); complexity not justified

#### Option B: Cilium Service Mesh
- **Pros:** eBPF-based (no sidecar), excellent performance, network policy integration
- **Cons:** Requires kernel 5.10+, less mature mesh features, complex CNI replacement
- **Why rejected:** Operational complexity of CNI replacement; kind cluster compatibility issues

#### Option C: No Mesh (Application-Level TLS)
- **Pros:** No additional infrastructure, full control
- **Cons:** Each service must implement TLS, certificate rotation, observability
- **Why rejected:** Doesn't scale; every new service needs TLS code

### Trade-offs

- Accept ~15MB memory overhead per pod for transparent mTLS
- Accept Linkerd's simpler traffic policies (vs Istio's VirtualService) for operational simplicity
- Maintain application-level circuit breaker alongside mesh (defense in depth)

## Performance Baseline

Benchmarks from spectre-proxy release build (2026-02-16, localhost, no mesh):

| Endpoint | RPS | p50 | p95 | p99 |
|----------|-----|-----|-----|-----|
| GET /health (50 conns) | 58,130 | 0.5ms | 2.8ms | 4.6ms |
| GET /metrics (auth, 50 conns) | 59,692 | 0.4ms | 2.6ms | 5.2ms |
| POST /ingest (auth+NATS, 50 conns) | 68,903 | 0.4ms | 2.1ms | 4.0ms |
| GET /health (200 conns) | 100,733 | 1.0ms | 6.3ms | 18.1ms |

**Memory**: 25.8 MB RSS, 13 threads

Linkerd published overhead: ~1ms p99 added latency per hop. Expected mesh overhead < 5% at current RPS levels.

## Consequences

### Positive

- Automatic mTLS for all pod-to-pod communication
- Per-route golden metrics without code changes (Linkerd Viz)
- Traffic splitting enabled for canary deployments
- Certificate rotation handled by Linkerd identity system

### Negative

- Additional control plane to manage (destination, identity, proxy-injector)
- NATS port must be explicitly skipped (protocol detection limitation)
- Debugging requires understanding both application and mesh layers

### Risks

| Risk | Probability | Impact | Mitigation |
|------|-------------|--------|------------|
| Sidecar breaks non-HTTP protocols | Medium | High | skip-outbound-ports for NATS (4222) |
| Control plane failure | Low | High | Linkerd proxy continues in fail-open mode |
| Version upgrades break mesh | Low | Medium | Pin Linkerd version, test upgrades in dev first |
