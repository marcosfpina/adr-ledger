## Context

### Discovery

During Solidity test validation on 2026-01-31, 18 out of 69 tests (26% failure rate) were failing with the error:
```
LGPD_Article7_ConsentRequired(0x...003)
```

### Root Cause Investigation

Deep forensic analysis revealed a critical bug in `contracts/src/LGPDConsent.sol:210`:

```solidity
// CURRENT (BUGGY):
function hasConsent(address dataSubject) internal view override returns (bool) {
    return hasConsent(dataSubject, msg.sender);  // BUG: msg.sender is the USER, not the contract
}

// CORRECT:
function hasConsent(address dataSubject) internal view override returns (bool) {
    return hasConsent(dataSubject, address(this));  // FIX: address(this) is the contract processor
}
```

### The Problem

In Solidity, `msg.sender` does NOT change during internal function calls. When a user calls `applyForLoan()`:

1. User grants consent: `grantConsent(address(lending), 365 days, "Loan")`
   - Storage: `consents[user][lending] = true` ✓ Correct location

2. User calls: `lending.applyForLoan()`
   - Modifier checks: `hasConsent(user, msg.sender)`
   - But `msg.sender` is still the user, not the lending contract!
   - Checks: `consents[user][user]` ✗ Wrong location (doesn't exist)
   - Result: REVERT

### Workaround Applied

Tests were changed to use "self-consent":
```solidity
lending.grantConsent(borrower, 365 days, "Loan");  // User consents to SELF
```

This makes tests pass but violates LGPD semantic intent: consent should be granted to the PROCESSOR (the contract), not to oneself.

### Scientific Validation

Created `ConsentBugValidation.t.sol` with 6 experiments, all passing:
- **Experiment 1**: Proves workaround works but semantically wrong
- **Experiment 2**: Proves original design fails with current bug
- **Experiment 3**: Demonstrates msg.sender vs address(this) difference
- **Experiment 4**: Storage visualization comparison
- **Experiment 5**: Simulates correct fix behavior
- **Experiment 6**: Side-by-side comparison of approaches

## Decision

**Apply the correct architectural fix**: Change `msg.sender` to `address(this)` in `LGPDConsent.sol:210`.

This is NOT optional - it is a **blocker for mainnet deployment** due to incorrect LGPD compliance semantics.

## Rationale

### Drivers

1. **LGPD Compliance**: Current workaround violates LGPD Art. 7 semantics
   - Correct: "User consents that PROCESSOR uses their data"
   - Current: "User consents that USER uses their own data" (nonsensical)

2. **Legal Risk**: Incorrect consent model could result in fines up to R$50M (2% of revenue)

3. **Architectural Integrity**: Design documents clearly state processor should be the contract
   ```solidity
   /// @param processor Address of the entity that will process data
   ```

4. **Production Readiness**: Cannot deploy to mainnet with semantic bugs

### Alternatives Considered

#### Option A: Keep Workaround (Rejected)
- **Pros:**
  - Tests already passing
  - No contract changes needed
  - Zero deployment cost
- **Cons:**
  - Violates LGPD semantics
  - Confuses future developers
  - Fails compliance audits
  - Cannot explain to regulators
- **Why rejected:** Unacceptable legal and architectural risk

#### Option B: Apply Correct Fix (RECOMMENDED)
- **Pros:**
  - Correct LGPD semantics
  - Matches original design intent
  - Maintainable long-term
  - Audit-ready
- **Cons:**
  - Requires contract redeploy
  - 35 minutes of work
- **Why chosen:** Only acceptable solution for production

### Trade-offs

- **Accept**: 35 minutes of effort + redeploy costs
- **Gain**: Legally compliant, semantically correct, production-ready system
- **Avoid**: Potential R$50M fines, audit failures, architectural debt
