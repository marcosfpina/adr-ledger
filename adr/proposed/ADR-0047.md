## Context

O projeto NEXUS/Neutron precisa gerenciar secrets sensíveis (API keys Anthropic, DeepSeek, JWT
signing key, database credentials) de forma segura. Durante o desenvolvimento inicial, a decisão
foi usar `.env` como solução temporária para desbloquear os testes de integração com LLMs reais.

O sistema NixOS host já possui infraestrutura SOPS completa via `sops-nix`:
- `/etc/nixos/secrets/api-keys.yaml` — secrets criptografados com age
- `/run/secrets/anthropic_api_key` — decriptado no boot com mode 0440
- `neutron/core/sops.py` — conector Python já implementado com fallback chain

O conector `neutron/core/sops.py` já está funcional e implementa a prioridade:
```
env var (shellHook/CI) → /run/secrets/ (sops-nix) → default
```

A integração com o `flake.nix` do projeto está pendente.

## Decision

Adotar SOPS + sops-nix como solução canônica de gestão de secrets para produção, com
`.env` permitido **apenas em desenvolvimento local** e **nunca comitado no repositório**.

A integração com o `flake.nix` do Neutron usará `sops exec-env` no `shellHook` para
injetar os secrets como variáveis de ambiente ao entrar no devShell, eliminando a
necessidade de `.env` mesmo em desenvolvimento.

```nix
# flake.nix — shellHook (implementação futura)
shellHook = ''
  if command -v sops &>/dev/null && [ -f /etc/nixos/secrets/api-keys.yaml ]; then
    _json=$(sops --decrypt --output-type json /etc/nixos/secrets/api-keys.yaml 2>/dev/null)
    export ANTHROPIC_API_KEY=$(echo "$_json" | jq -r '.anthropic_api_key // ""')
    export DEEPSEEK_API_KEY=$(echo "$_json" | jq -r '.deepseek_api_key // ""')
    unset _json
    echo "✓ SOPS secrets carregados"
  fi
'';
```

## Rationale

### Drivers

- Secrets em `.env` são inseguros em produção — ficheiros sem criptografia em disco
- O stack já é NixOS declarativo — SOPS é a extensão natural para secrets
- LGPD exige rastreabilidade de acesso a credenciais usadas no processamento de dados pessoais
- `sops-nix` já está operacional no host — sem overhead de nova infraestrutura

### Alternatives Considered

#### Option A: `.env` file (rejeitado para produção)
- **Pros:** Zero setup, funciona imediatamente
- **Cons:** Sem criptografia, risco de commit, sem auditoria
- **Why rejected:** Aceitável apenas como workaround temporário em dev

#### Option B: HashiCorp Vault (futuro)
- **Pros:** Dynamic secrets, lease management, auditoria completa
- **Cons:** Overhead operacional alto, requer infra adicional
- **Why rejected:** Candidato para pós-Series A quando a complexidade justificar

#### Option C: AWS Secrets Manager (rejeitado)
- **Pros:** Managed service, rotação automática
- **Cons:** Cloud lock-in, fora do stack declarativo NixOS, custo
- **Why rejected:** Contra o princípio de infraestrutura hermética do projeto

### Trade-offs

- sops-nix não tem hot reload — mudanças de secret requerem `nixos-rebuild switch`
- Dev local precisa de setup inicial de age key (documentação necessária)

## Consequences

### Positive

- Zero secrets em texto plano em disco em produção
- Auditoria de acesso via logs do sops-nix
- Integração transparente — `neutron/core/sops.py` já funcional
- CI/CD via `sops exec-env` em runners efêmeros sem estado persistente

### Negative

- Rotação de secrets em produção requer `nixos-rebuild switch` (não instantâneo)
- Novos devs precisam configurar age key local para acessar secrets

### Risks

| Risk | Probability | Impact | Mitigation |
|------|-------------|--------|------------|
| Perda da age key SSH do host | low | critical | Backup em `/etc/nixos/sec/` + recovery key |
| Permissões incorretas em `/run/secrets/` | low | high | sops-nix configura 0440 automaticamente |
| Dev commita `.env` acidentalmente | medium | high | `.gitignore` + pre-commit hook |
