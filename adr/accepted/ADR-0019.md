## Context

Cerebro needed to evolve from a basic CLI tool with 15 scattered scripts into a unified platform with three interfaces: TUI (interactive), CLI (scriptable), and GUI (visual). The primary challenge was choosing a framework for building a modern, performant Terminal User Interface that could:

- Handle large datasets efficiently (1000+ projects)
- Provide rich interactive visualizations
- Integrate seamlessly with existing async FastAPI backend
- Support keyboard shortcuts and mouse interaction
- Work reliably over SSH
- Maintain fast startup times (<2s)

The TUI would serve as the default interface for interactive terminal sessions, complementing the CLI for scripting and GUI dashboard for web-based visualization.

## Decision

**Adopt Textual 0.47+ as the TUI framework for Cerebro's interactive terminal interface.**

Textual will power 6 specialized screens (Dashboard, Projects, Intelligence, Scripts, GCP Credits, Logs) with rich widgets, async event handling, and CSS-like styling.

## Rationale

### Drivers

- Need for modern TUI framework to replace basic CLI
- Requirement for async/await compatibility with existing backend
- Large dataset handling (lazy loading, virtual scrolling)
- Cross-platform support including SSH environments
- Active maintenance and community support
- Developer experience (hot reload, debugging tools)

### Alternatives Considered

#### Option A: curses (stdlib)
- **Pros:** No dependencies, mature, stable
- **Cons:** Low-level API, complex event handling, no modern widgets, synchronous only, poor cross-platform support
- **Why rejected:** Would require building entire widget system from scratch, no async support

#### Option B: blessed
- **Pros:** Simpler than curses, better terminal compatibility
- **Cons:** Limited widget library, outdated (last release 2018), no async support, poor documentation
- **Why rejected:** Unmaintained project, lacks modern features needed for rich UI

#### Option C: rich (display-only)
- **Pros:** Already used in Cerebro, excellent rendering, great documentation
- **Cons:** Not interactive (no event loop), display-only library
- **Why rejected:** Cannot build interactive applications, would need separate framework for input handling

#### Option D: prompt_toolkit
- **Pros:** Mature, used by IPython, good input handling
- **Cons:** Focused on REPL/prompt use cases, limited layout system, complex application structure
- **Why rejected:** Not designed for full-screen applications with multiple screens

### Trade-offs

- **Accepted:** External dependency (~500KB) vs stdlib-only approach
- **Accepted:** Learning curve for Textual API vs familiar curses
- **Accepted:** Framework lock-in vs building custom solution
- **Mitigated:** Lazy import strategy to avoid loading TUI code in CLI-only usage

## Consequences

### Positive

- **Modern DX:** Hot reload, CSS-like styling, component reusability
- **Performance:** Built-in virtual scrolling, lazy rendering, async-native
- **Rich Widgets:** DataTable, Tree, Input, Log, Sparkline out of box
- **Active Development:** 40k+ GitHub stars, regular updates, responsive maintainers
- **Cross-platform:** Works on Linux, macOS, Windows, via SSH
- **Debugging:** Built-in devtools, console output redirection
- **Integration:** Seamless async/await with FastAPI backend

### Negative

- **Dependency Size:** +500KB to package size
- **Learning Curve:** Team must learn Textual-specific patterns (reactive properties, message passing)
- **Version Lock:** Must track breaking changes in 0.x releases
- **Lazy Loading Complexity:** Requires careful import management to preserve CLI startup speed

### Risks

| Risk | Probability | Impact | Mitigation |
|------|-------------|--------|------------|
| Framework abandonment | Low | High | Active community, commercial backing (Textualize.io), fork-able if needed |
| Breaking changes in 0.x | Medium | Medium | Pin to minor version, comprehensive tests, monitor changelog |
| Performance issues with large datasets | Low | High | Implemented lazy loading, pagination, virtual scrolling |
| CLI startup slowdown | Medium | Medium | Lazy imports pattern (defer textual import until TUI invoked) |
