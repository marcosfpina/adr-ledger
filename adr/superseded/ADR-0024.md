## Context

The Phantom ecosystem has evolved into multiple repositories with overlapping responsibilities:

| Repository | Current State | Purpose |
|------------|---------------|---------|
| `phantom/` | Active, production-ready | Judge API, Cerebro RAG, SPECTRE NLP |
| `phantom-stack/` | Partially active | Contains useful components + orphan files |
| `phantom-soc/` | Future project | Security Operations Center capabilities |

This fragmentation causes:
- Confusion about which repository is the "source of truth"
- Orphan files scattered without documentation
- Unclear production deployment target
- Risk of duplicated effort across repositories

## Decision

We will consolidate the Phantom ecosystem with the following strategy:

### 1. `phantom/` → Production Target (👑 REI)
- Designate as the canonical Phantom implementation
- Validate and deploy to production
- All active development focuses here

### 2. `phantom-stack/` → Absorb & Archive (📦)
- Inventory all useful components
- Absorb valuable code into `phantom/`
- Document orphan files with explicit paths for future reference
- Archive the repository after extraction complete

### 3. `phantom-soc/` → Isolate for Future (🔮)
- Mark as future SOC project
- No active development until Phantom core is production-stable
- Preserve existing work for future integration

## Rationale

### Drivers

1. **Production urgency**: Phantom Judge API needs to reach production for the intelligence loop
2. **Clarity**: Single canonical repository eliminates confusion
3. **Documentation**: Orphan files need explicit paths for workflow efficiency
4. **Focus**: SOC features can wait until core is stable

### Alternatives Considered

#### Option A: Merge All Into Monorepo
- **Pros:** Single codebase, simpler navigation
- **Cons:** Loses separation of concerns, SOC features not ready for integration
- **Why rejected:** Premature consolidation would slow production deployment

#### Option B: Keep All Repositories Active
- **Pros:** No migration effort required
- **Cons:** Continued confusion, duplicated development effort
- **Why rejected:** Technical debt accumulation outweighs short-term convenience

### Trade-offs

- Accept short-term effort to archive `phantom-stack/` for long-term clarity
- Delay SOC features to maintain focus on production Phantom

## Consequences

### Positive

- Clear production target: `phantom/` is canonical
- Reduced contributor confusion
- Documented orphan files enable systematic cleanup
- SOC capabilities preserved for future development

### Negative

- `phantom-stack/` archive loses some active view of development history
- SOC features delayed until Phase 4+

### Risks

| Risk | Probability | Impact | Mitigation |
|------|-------------|--------|------------|
| Useful code missed during absorption | Low | Medium | Create detailed inventory before archiving |
| Orphan file documentation incomplete | Low | Low | Use grep/find to systematically scan |
| SOC delay impacts future roadmap | Low | Medium | Document clear activation criteria |
