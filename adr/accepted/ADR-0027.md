# ADR-0027: Phantom Phase 1 - Backend API Implementation with Real Data Integration

## Context

Phantom v2.0.0 is a local-first AI document intelligence framework with three interfaces:
- CLI (Typer)
- REST API (FastAPI)
- Desktop UI (Tauri + SvelteKit)

Prior to this work, the desktop UI (`cortex-desktop/`) was fully implemented but non-functional because the backend API endpoints returned empty/stub data. All backend infrastructure existed (CORTEX processor, FAISS vector store, embedding generator, LLM provider) but was not exposed via REST API.

### Problem Statement

**Frontend called 7 endpoints that either didn't exist or returned stub data:**
1. `POST /process` - Document processing (returned TODO comment)
2. `POST /vectors/search` - Vector search (didn't exist)
3. `POST /vectors/index` - Document indexing (didn't exist)
4. `POST /api/chat` - RAG chat (didn't exist)
5. `GET /api/models` - Models listing (didn't exist)
6. `POST /api/prompt/test` - Prompt testing (didn't exist)
7. `GET /api/system/metrics` - System metrics (didn't exist)

**Impact:**
- Desktop UI appeared broken
- No way to test real document processing
- Vector search functionality inaccessible
- RAG chat not functional
- System resource monitoring unavailable

## Decision

**Implement all 7 Phase 1 endpoints with real data integration.**

This includes:
- Wiring up existing backend services (CORTEX, FAISS, embeddings, LLM)
- Creating REST API endpoints with proper error handling
- Adding Pydantic validation for all requests/responses
- Implementing singleton pattern for resource efficiency
- Writing comprehensive integration tests (30+)
- Ensuring frontend compatibility with TypeScript interfaces

## Implementation Details

### Architecture Pattern: Global Singletons

```python
# Module-level instances shared across requests
_embedding_generator = None  # sentence-transformers model
_vector_store = None         # FAISS index

def get_embedding_generator():
    global _embedding_generator
    if _embedding_generator is None:
        _embedding_generator = EmbeddingGenerator()
        logger.info("Initialized EmbeddingGenerator")
    return _embedding_generator

def get_vector_store():
    global _vector_store
    if _vector_store is None:
        embedder = get_embedding_generator()
        _vector_store = FAISSVectorStore(embedding_dim=embedder.dimension)
        logger.info(f"Initialized FAISSVectorStore (dim={embedder.dimension})")
    return _vector_store
```

**Rationale:**
- Models are expensive to load (sentence-transformers ~500MB)
- FAISS index needs to persist across requests for vector search to work
- Singleton pattern reduces memory usage and improves response times

### RAG Pipeline Implementation

```
User Query
    ↓
Embedding Generation (sentence-transformers)
    ↓
Vector Search (FAISS top-k cosine similarity)
    ↓
Context Retrieval (5 most similar documents by default)
    ↓
Prompt Construction:
    System Prompt
    + Conversation History (last 5 messages)
    + Retrieved Context
    + Current Query
    ↓
LLM Generation (llama.cpp @ port 8081)
    ↓
Response + Source Citations
```

```python
def test_vector_index_and_search(self, client):
    # Step 1: Index a document
    content = b"Python is a programming language for AI."
    resp1 = client.post(
        "/vectors/index",
        files={"file": ("python.txt", content, "text/plain")}
    )
    assert resp1.status_code == 200
    assert resp1.json()["chunks_indexed"] > 0

    # Step 2: Search for it
    resp2 = client.post(
        "/vectors/search?query=programming+language&top_k=3"
    )
    assert resp2.status_code == 200
    results = resp2.json()["results"]
    assert len(results) > 0
    assert "programming" in results[0]["text"].lower()
```

## Consequences

### Positive

1. **Frontend Fully Functional**
   - Desktop UI can now process documents, search vectors, and chat with RAG
   - All tabs (Process, Search, Chat, Workbench, Settings) work with real data

2. **Zero Mock Data**
   - All responses come from actual services (CORTEX, FAISS, psutil, llama.cpp)
   - System metrics show real host CPU, memory, disk, and VRAM usage

3. **Production-Ready Quality**
   - Pydantic validation prevents malformed requests
   - Proper HTTP status codes (200, 400, 422, 500)
   - Comprehensive error handling and logging
   - 30+ integration tests ensure reliability

4. **Efficient Resource Usage**
   - Singleton pattern: Models loaded once, reused forever
   - No unnecessary model reloading between requests
   - Reduced memory footprint

5. **Frontend-Backend Alignment**
   - TypeScript interfaces in `api.ts` match Python Pydantic models
   - All expected endpoints now exist and return correct schemas

### Negative

1. **Vector Store Not Persistent**
   - **Problem:** In-memory FAISS index lost on server restart
   - **Impact:** Indexed documents disappear, requiring re-indexing
   - **Mitigation:** Add save/load endpoints in Phase 2

2. **Thread Safety Issues**
   - **Problem:** Global singletons not thread-safe for concurrent writes
   - **Impact:** Rare race conditions during vector store modifications
   - **Mitigation:** Add threading locks for write operations

3. **LLM Availability Not Checked**
   - **Problem:** Chat endpoint assumes llama.cpp is running
   - **Impact:** Failures if LLM service is down (fallback message shown)
   - **Mitigation:** Add health check before chat requests

4. **Token Estimation Approximate**
   - **Problem:** Simple calculation (4 chars = 1 token) is inaccurate
   - **Impact:** Prompt test endpoint returns rough estimates
   - **Mitigation:** Integrate tiktoken library for accurate counting

### Risks

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| Memory exhaustion from large vector store | Medium | High | Add monitoring, implement cleanup policy |
| Concurrent modification race conditions | Low | Medium | Add threading locks for writes |
| LLM downtime breaks chat | Medium | Medium | Implement health checks, better fallbacks |
| FAISS index corruption | Low | High | Add save/backup functionality |
