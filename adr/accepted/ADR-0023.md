## Context

Cerebro's TUI needed to provide comprehensive access to all system functionality in an intuitive, navigable interface. The system has multiple distinct operational domains:

1. **System Overview:** Health metrics, alerts, recent activity
2. **Project Management:** Browse, analyze, and summarize 1000+ projects
3. **Intelligence Queries:** Interactive RAG/search with result visualization
4. **Script Execution:** Launch 24 CLI commands with parameter forms
5. **Resource Management:** GCP credit tracking and batch operations
6. **Observability:** Real-time log streaming and filtering

**Design Constraints:**
- Terminal real estate is limited (80x24 minimum support)
- Users need quick navigation between domains
- Long-running operations need progress feedback
- Some operations are related (e.g., scan projects → view in Projects screen)
- Keyboard-first navigation for power users

**Alternative Approaches Considered:**
- **Single Screen:** Too cramped, context switching difficult
- **Many Small Screens:** Too fragmented, hard to navigate
- **Dynamic Screen Creation:** Complex, no stable navigation

## Decision

**Implement 6 specialized screens with sidebar navigation, optimized for distinct workflows.**

**Screen Architecture:**

```
TUI Application
├── MainMenu (Sidebar) - Always visible navigation
├── DashboardScreen - System overview, quick actions
├── ProjectsScreen - Project DataTable, analysis
├── IntelligenceScreen - Query input, results display
├── ScriptsScreen - Command launcher with forms
├── GCPCreditsScreen - Credit tracking, batch burn
└── LogsScreen - Live log tail with filtering
```

**Navigation Model:**
- Persistent sidebar (1-6 number keys + q to quit)
- Each screen full-featured for its domain
- Command Router bridges TUI actions to CLI functions
- WebSocket for real-time updates (alerts, progress)

## Rationale

### Drivers

- **Domain Separation:** Each screen focused on single concern
- **Keyboard Navigation:** Number keys for instant screen switching
- **Discoverability:** Sidebar shows all available screens
- **Real Estate:** Dedicated screen allows rich visualizations
- **Workflow Support:** Natural task boundaries (query → view projects → check logs)
- **Maintainability:** Clear code organization (one file per screen)

### Alternatives Considered

#### Option A: Single Dashboard Screen
- **Pros:** Simple, everything visible at once
- **Cons:** Cramped, poor information density, can't show details
- **Why rejected:** 80x24 terminal can't fit project table + logs + metrics + controls

#### Option B: Modal/Dialog System
- **Pros:** Single main screen, pop-ups for actions
- **Cons:** Poor for terminals, hard to navigate, lose context
- **Why rejected:** Dialogs work poorly in TUI, can't see background while modal open

#### Option C: Nested Screens (tree hierarchy)
- **Pros:** Logical grouping, unlimited depth
- **Cons:** Deep navigation, breadcrumb confusion, back button complexity
- **Why rejected:** Too many levels to navigate, slow to reach deep screens

#### Option D: Tab Interface (browser-style)
- **Pros:** Familiar metaphor, multiple concurrent contexts
- **Cons:** Tab bar takes space, state management complexity, less keyboard-friendly
- **Why rejected:** Sidebar with number keys faster than tab switching

#### Option E: Split Panes (tmux-style)
- **Pros:** See multiple views simultaneously
- **Cons:** Each pane too small, complex layout management, keyboard navigation harder
- **Why rejected:** Terminal resolution insufficient for meaningful splits

### Trade-offs

- **Accepted:** 6 screens vs single view (navigation overhead vs information density)
- **Accepted:** Specialized screens vs generic container (maintainability vs flexibility)
- **Accepted:** Command Router layer vs direct backend calls (indirection vs reusability)
- **Benefit:** Each screen optimized for specific workflow

## Consequences

### Positive

- **Clear Organization:** Each screen has single responsibility
- **Rich Visualizations:** Full screen for DataTables, logs, progress bars
- **Fast Navigation:** Number keys 1-6 for instant switching
- **Maintainable Code:** One file per screen, clear boundaries
- **Extensible:** Easy to add new screens without touching existing ones
- **Keyboard-First:** Power users can work without mouse
- **Consistent UX:** All screens share navigation pattern

### Negative

- **Navigation Overhead:** Must switch screens to access features
- **Context Switching:** Can't see multiple domains simultaneously
- **Maintenance Burden:** 6 screens to keep functional
- **State Management:** Must handle screen-specific state
- **Command Router Complexity:** Additional abstraction layer
- **Memory Usage:** All screens loaded in memory (mitigated by lazy loading)

### Risks

| Risk | Probability | Impact | Mitigation |
|------|-------------|--------|------------|
| Users forget which screen has feature | Medium | Low | Clear sidebar labels, status bar hints, help screen |
| Navigation feels slow | Low | Medium | Fast screen switching (Textual is instant), number key shortcuts |
| Screen state gets stale | Medium | Medium | WebSocket real-time updates, refresh on screen enter |
| Too many screens (feature creep) | Medium | Medium | Maintain 6-screen limit, group related features |
| Command Router becomes complex | Low | Medium | Keep routing simple (direct function calls), document patterns |

## Implementation

### Screen Details

**1. DashboardScreen** (`tui/screens/dashboard.py`)
- **Purpose:** System overview and quick actions
- **Widgets:** Health metrics panel, alert list, recent activity log, action buttons
- **Actions:** Scan projects, query intelligence, run scripts
- **Updates:** WebSocket for live metrics

**2. ProjectsScreen** (`tui/screens/projects.py`)
- **Purpose:** Browse and manage project portfolio
- **Widgets:** Sortable DataTable (name, health, last scan), detail panel, action buttons
- **Features:** Filter, search, sort, drill-down to project details
- **Actions:** Analyze, summarize, open in file manager
- **Performance:** Virtual scrolling for 1000+ projects

**3. IntelligenceScreen** (`tui/screens/intelligence.py`)
- **Purpose:** Interactive RAG/intelligence queries
- **Widgets:** Query input with history, results display with syntax highlighting, filter panel
- **Features:** Query suggestions, result export, source linking
- **Performance:** Streaming results display

**4. ScriptsScreen** (`tui/screens/scripts.py`)
- **Purpose:** Launch any of 24 CLI commands
- **Widgets:** Command tree by group, parameter forms, progress bars, result display
- **Features:** Command search, recent commands, output capture
- **Integration:** Command Router for execution

**5. GCPCreditsScreen** (`tui/screens/gcp_credits.py`)
- **Purpose:** Monitor and consume GCP credits
- **Widgets:** Credit balance gauge, batch burn controls, query history, cost calculator
- **Features:** Real-time burn progress, consumption graphs
- **Safety:** Confirmation dialogs for large burns

**6. LogsScreen** (`tui/screens/logs.py`)
- **Purpose:** Real-time log monitoring
- **Widgets:** Log view with auto-scroll, filter controls (level, module), export button
- **Features:** Live tail, search, highlight, pause/resume
- **Performance:** Ring buffer for memory efficiency

### Command Router

**Purpose:** Bridge TUI actions to CLI command functions

```python
# tui/commands/router.py
class CommandRouter:
    """Route TUI actions to CLI command implementations."""

    async def run_command(self, group: str, command: str, **kwargs):
        """Execute CLI command from TUI context."""
        # Lazy import to avoid circular dependencies
        if group == "gcp":
            from phantom.commands.gcp import batch_burn
            async for progress in batch_burn(**kwargs):
                yield progress  # Stream to TUI progress bar

    async def query_intelligence(self, query: str):
        from phantom.commands.knowledge import query
        return await query(query)

    # ... other routing methods
```

**Benefits:**
- Reuses CLI command logic (no duplication)
- Provides async/streaming interface for TUI
- Centralizes TUI-to-CLI integration

**Keyboard Shortcuts:**

- `?`: Help screen
