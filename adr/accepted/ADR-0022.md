## Context

Cerebro started with a basic Typer CLI (`cerebro`) with 9 commands in 3 groups (knowledge, rag, ops). Over time, 15 additional Python scripts accumulated in `scripts/`:

**GCP Management (3):**
- `batch_burn.py` - Burn GCP credits via batch queries
- `monitor_credits.py` - Track credit consumption
- `create_search_engine.py` - Configure Google Search Engine

**Strategy/Intelligence (4):**
- `strategy_optimizer.py` - Optimize strategic planning
- `salary_intel.py` - Salary intelligence gathering
- `personal_moat_builder.py` - Personal competitive advantage
- `trend_predictor.py` - Trend prediction

**Content (1):**
- `content_gold_miner.py` - Content mining

**Knowledge (4):**
- `generate_queries.py` - Query generation
- `index_repository.py` - Repository indexing
- `etl_docs.py` - Document ETL
- `generate_docs.py` - Documentation generation

**Testing (3):**
- `grounded_search.py` - Google Search API testing
- `grounded_generation_test.py` - Grounded generation testing
- `verify_grounded_api.py` - API verification

**Problems with scattered scripts:**

1. **Discoverability:** Users must know scripts exist, no `--help` listing
2. **Inconsistent UX:** Each script has different arg parsing (argparse vs typer vs custom)
3. **No Integration:** Scripts can't easily call each other or share state
4. **Duplication:** Common code (config loading, logging) duplicated
5. **Testing:** Harder to test standalone scripts vs CLI commands
6. **Documentation:** 15 separate README sections vs unified help system
7. **Maintenance:** Changes require updating multiple files

## Decision

**Consolidate all 15 scripts into Typer command groups, creating unified `cerebro` CLI with 24 total commands.**

**New structure:**
```
cerebro (24 commands in 7 groups)
├── knowledge (9 cmds) - 5 existing + 4 migrated
├── rag (3 cmds) - existing
├── ops (1 cmd) - existing
├── gcp (3 cmds) - NEW: burn, monitor, create-engine
├── strategy (4 cmds) - NEW: optimize, salary, moat, trends
├── content (1 cmd) - NEW: mine
└── test (3 cmds) - NEW: grounded-search, grounded-gen, verify-api
```

**Keep original scripts with deprecation warning** for backward compatibility during transition.

## Rationale

### Drivers

- **UX Consistency:** Single `cerebro --help` shows all capabilities
- **Discoverability:** Users find commands through help system
- **Maintainability:** Centralized code, shared utilities
- **Testing:** Unit test commands directly
- **Integration:** Commands can call each other easily
- **Documentation:** Auto-generated help from Typer docstrings
- **Type Safety:** Typer provides type checking and validation

### Alternatives Considered

#### Option A: Keep Scripts Separate
- **Pros:** No migration effort, isolated code
- **Cons:** Poor UX, fragmentation, duplication, hard to maintain
- **Why rejected:** Technical debt accumulating, user confusion

#### Option B: Wrapper Script (cerebro-scripts)
- **Pros:** Minimal changes, preserves scripts
- **Cons:** Still fragmented, doesn't solve underlying issues, adds another layer
- **Why rejected:** Band-aid solution, doesn't address root problems

#### Option C: Plugin System
- **Pros:** Extensible, clean boundaries
- **Cons:** Over-engineering, runtime discovery complexity, slower startup
- **Why rejected:** Overkill for 15 scripts, adds unnecessary complexity

#### Option D: Separate CLI Binary per Group
- **Pros:** Independent binaries, clear separation
- **Cons:** User confusion (cerebro vs cerebro-gcp?), installation complexity, code duplication
- **Why rejected:** Violates unified platform goal

### Trade-offs

- **Accepted:** Migration effort (refactor 15 scripts) vs long-term maintainability
- **Accepted:** Breaking changes for script users vs improved UX
- **Accepted:** Single codebase vs modular packages
- **Mitigated:** Keep scripts with deprecation warnings during transition

## Consequences

### Positive

- **Unified Interface:** Single `cerebro` command with consistent UX
- **Discoverability:** `cerebro --help` lists all 24 commands
- **Better Help:** Rich help text from Typer docstrings
- **Type Safety:** Typer validates arguments at CLI boundary
- **Testability:** Commands are functions, easy to unit test
- **Code Reuse:** Shared utilities, config loading, error handling
- **Integration:** Commands can import and call each other
- **Documentation:** Auto-generated from code, always in sync
- **Maintainability:** Single codebase, consistent patterns

### Negative

- **Migration Effort:** Refactor 15 scripts (~2-3 days work)
- **Breaking Changes:** Script users must update workflows
- **Import Time:** Slight increase if all commands imported eagerly (mitigated by lazy imports)
- **Testing Burden:** Must test 24 commands instead of 9
- **Monolithic CLI:** All code in one package (trade-off for simplicity)

### Risks

| Risk | Probability | Impact | Mitigation |
|------|-------------|--------|------------|
| Break existing script users | High | Medium | Keep scripts with deprecation warning, document migration |
| Introduce bugs during refactor | Medium | High | Preserve original script logic, comprehensive tests, gradual migration |
| CLI becomes too complex | Low | Medium | Good organization (7 groups), clear naming, rich help text |
| Performance regression | Low | Low | Lazy imports for heavy dependencies, benchmarking |
| Command name conflicts | Low | High | Careful naming, namespace with groups |

## Implementation

### Migration Pattern

**Before (script):**
```python
# scripts/batch_burn.py
import argparse

def main(args):
    # Logic here
    pass

if __name__ == "__main__":
    parser = argparse.ArgumentParser()
    parser.add_argument("--queries", type=int, default=100)
    args = parser.parse_args()
    main(args)
```

**After (command):**
```python
# src/phantom/commands/gcp.py
import typer

gcp_app = typer.Typer(help="GCP resource management")

@gcp_app.command("burn")
def batch_burn(
    queries: int = typer.Option(100, help="Number of queries to execute"),
    workers: int = typer.Option(10, help="Concurrent workers")
):
    """Execute batch query burn for GCP credit consumption."""
    # Logic migrated from script (preserved exactly)
    pass
```

**Deprecation wrapper:**
```python
# scripts/batch_burn.py (kept for backward compat)
import warnings
warnings.warn(
    "batch_burn.py is deprecated. Use: cerebro gcp burn",
    DeprecationWarning
)
from phantom.commands.gcp import batch_burn
# Forward to new implementation
```
