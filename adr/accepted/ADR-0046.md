## Context

O CHAINSCOPE coleta dados de fontes heterogêneas (DefiLlama, CoinGecko, The Graph,
Discourse forums, etc.), cada uma com estrutura de resposta diferente. Era necessário
definir como converter `RawDataPoint` (dado bruto, com `raw_payload` opaco) em
linhas estruturadas para `protocol_metrics` no TimescaleDB.

A decisão impacta testabilidade, extensibilidade e separação de responsabilidades
no pipeline de coleta.

## Decision

**Padrão mapper/engine com duas camadas distintas:**

1. **Mappers** (`services/normalizer/mappers/{source}_mapper.py`): funções puras,
   sem IO, que recebem um dict/list de response e retornam `list[dict]` de rows.

2. **NormalizerEngine** (`services/normalizer/engine.py`): dispatch por `source_id`
   + `metadata["endpoint"]`, chama o mapper correto, captura exceções, loga erros.

```python
# Uso:
engine = NormalizerEngine()
rows = engine.normalize(dp)              # nunca levanta exceção
await client.upsert_metrics_batch(rows)  # rows prontos para TimescaleDB
```

```python
# Adicionar nova fonte:
# 1. Criar services/normalizer/mappers/coingecko_mapper.py
# 2. Adicionar ao dispatch em engine.py:
#    "coingecko": self._normalize_coingecko
```

## Rationale

### Drivers

- Fontes heterogêneas exigem mapeamento específico por fonte
- Funções puras são testáveis sem mocks de HTTP ou banco
- Dispatch por dict é O(1) e extensível sem modificar código existente
- `normalize()` não deve propagar exceções — pipeline deve ser resiliente

### Alternatives Considered

#### Option A: Normalização inline no adapter
- **Pros:** Menos arquivos, tudo em um lugar
- **Cons:** Acoplamento forte, impossível testar sem HTTP, viola SRP
- **Why rejected:** Adapter deve coletar; normalização é responsabilidade separada

#### Option B: Herança (NormalizerBase → DefiLlamaNormalizer)
- **Pros:** OOP clássico
- **Cons:** Herança profunda ao escalar, difícil composição de fontes
- **Why rejected:** Funções puras + dispatch são mais simples e Pythônicas

### Trade-offs

- Dois arquivos por fonte em vez de um — aceitável pela clareza de responsabilidade

## Consequences

### Positive

- Unit tests de mappers sem qualquer mock (apenas `datetime.now()` como input)
- `NormalizerEngine.normalize()` nunca levanta exceção — pipeline nunca trava por dados malformados
- Extensão linear: nova fonte = novo arquivo mapper + 1 linha no dispatch

### Negative

- Overhead inicial de dois arquivos por fonte (mapper + engine update)

### Risks

| Risk | Probability | Impact | Mitigation |
|------|-------------|--------|------------|
| Schema drift entre mapper e TimescaleDB | low | high | Nomes canônicos definidos em schemas.py |
