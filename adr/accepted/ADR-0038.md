## Context

During Phase 3 validation (Task #41: E2E Trace Propagation), we deployed Jaeger (all-in-one:1.53) and configured `spectre-proxy` to export traces via OpenTelemetry OTLP.

The default gRPC/tonic exporter (`opentelemetry_otlp::new_exporter().tonic()`) initialized without error and the batch processor ran normally, but **zero traces reached Jaeger**. No errors were logged — the batch exporter silently dropped all spans.

Root cause investigation:
1. Confirmed Jaeger OTLP HTTP endpoint (port 4318) accepts traces correctly via manual `curl`
2. Confirmed Jaeger OTLP gRPC endpoint (port 4317) is listening
3. Confirmed `tracing-opentelemetry` layer creates spans (visible in `tower_http::trace` debug logs)
4. The tonic 0.9.2 (pulled by opentelemetry-otlp) and workspace tonic 0.11 coexist but may cause subtle runtime issues
5. The batch exporter's async error path drops errors without surfacing them

## Decision

Switch the default OTLP transport from **gRPC/tonic** to **HTTP/protobuf** for trace export, while keeping gRPC available as a runtime option.

**Configuration:**
- Default: `OTEL_EXPORTER_OTLP_PROTOCOL=http/protobuf` (port 4318)
- Optional: `OTEL_EXPORTER_OTLP_PROTOCOL=grpc` (port 4317)
- Endpoint: `OTEL_EXPORTER_OTLP_ENDPOINT=http://<host>:<port>`

## Rationale

### Drivers

1. **Reliability**: HTTP/protobuf worked immediately; gRPC/tonic failed silently
2. **Debuggability**: HTTP traffic is inspectable with standard tools
3. **Compatibility**: Avoids tonic version conflicts in the workspace
4. **Standards compliance**: `OTEL_EXPORTER_OTLP_PROTOCOL` is the standard OpenTelemetry env var

### Alternatives Considered

#### Option A: gRPC/tonic (default)
- **Pros:** Lower overhead with HTTP/2 multiplexing, standard in OTLP ecosystem
- **Cons:** Failed silently in our setup, tonic version conflict potential
- **Why rejected:** Reliability is non-negotiable for observability infrastructure

#### Option B: HTTP/JSON
- **Pros:** Human-readable payloads
- **Cons:** 5-10x larger payloads than protobuf, higher serialization cost
- **Why rejected:** Protobuf is more efficient and equally supported

### Trade-offs

- Accept slightly higher per-request overhead (HTTP/1.1 vs gRPC/HTTP2) for guaranteed delivery
- Accept reqwest dependency for HTTP transport
- Keep gRPC path for future use when tonic version alignment is resolved

## Consequences

### Positive

- Traces reliably exported to Jaeger/Tempo collectors
- Runtime-configurable transport via standard OTEL env vars
- Global error handler catches any future silent export failures
- No code changes needed to switch backends in different environments

### Negative

- Additional `reqwest` dependency (already transitively present)
- HTTP/1.1 connection-per-request overhead at extreme scale (>10k spans/sec)

### Risks

| Risk | Probability | Impact | Mitigation |
|------|-------------|--------|------------|
| HTTP overhead at scale | low | low | Switch to gRPC when tonic versions align |
| reqwest security vuln | low | medium | Dependabot + cargo audit in CI |

## Implementation

### Files Changed

- `Cargo.toml` — `opentelemetry-otlp = { version = "0.14", features = ["metrics", "http-proto", "reqwest-client"] }`
- `crates/spectre-observability/src/lib.rs` — dual HTTP/gRPC exporter selection, global error handler

### Configuration

```bash
# Dev (100% sampling, HTTP)
OTEL_EXPORTER_OTLP_ENDPOINT=http://localhost:4318
OTEL_EXPORTER_OTLP_PROTOCOL=http/protobuf
OTEL_TRACES_SAMPLER_ARG=1.0

# Prod (10% sampling, can use gRPC if collector supports it)
OTEL_EXPORTER_OTLP_ENDPOINT=http://tempo:4318
OTEL_EXPORTER_OTLP_PROTOCOL=http/protobuf
OTEL_TRACES_SAMPLER_ARG=0.1
```
