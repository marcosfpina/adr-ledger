## Context

O CHAINSCOPE coleta dados de ~30+ fontes externas simultaneamente. Fontes públicas
como DefiLlama e CoinGecko têm rate limits não documentados, e falhas em uma fonte
não devem propagar para o restante do pipeline.

Era necessário definir um mecanismo uniforme de proteção antes de implementar
qualquer adapter de produção.

## Decision

**Usar `RateLimitedMixin` (sobre `spider_nix.AdaptiveRateLimiter` +
`spider_nix.CircuitBreaker`) como camada de proteção universal para adapters REST.**

```python
# Padrão de herança:
class DefiLlamaAdapter(RateLimitedMixin, BaseAdapter):
    def __init__(self):
        self.__init_rate_limiter__(initial_delay_ms=300.0)

    async def collect(self, params):
        await self._acquire()                              # aguarda delay adaptativo
        result = await self._call_protected(self._do_get, path)  # circuit breaker
        await self._report(result.status_code, elapsed_ms)       # ajusta rate limiter
        yield RawDataPoint(...)
```

Implementado em: `services/collector/adapters/rate_limited_mixin.py`

## Rationale

### Drivers

- Rate limits mudam sem aviso → delay adaptativo é mais robusto que fixo
- Falhas em cascata sem circuit breaker travam todo o pipeline
- spider-nix já tem `AdaptiveRateLimiter` e `CircuitBreaker` testados
- Mixin elimina duplicação — qualquer adapter herda com 1 linha

### Alternatives Considered

#### Option A: tenacity + asyncio.sleep fixo
- **Pros:** Zero dependência extra, simples
- **Cons:** Delay estático desperdiça capacidade; sem circuit breaker; duplicação
- **Why rejected:** Ineficiente e sem proteção a falhas em cascata

#### Option B: Rate limiter centralizado (Redis token bucket)
- **Pros:** Multi-process/node
- **Cons:** Redis overhead por request; overkill para single-process FASE 1
- **Why rejected:** Desnecessário agora — Redis disponível para FASE 3 quando escalar

### Trade-offs

- Dependência de spider-nix (biblioteca interna) — autor compartilhado, risco baixo
- Ordem MRO na herança múltipla: `(RateLimitedMixin, BaseAdapter)` — documentada

## Consequences

### Positive

- `AdaptiveRateLimiter` ajusta delay baseado em latência real e status codes
- `CircuitBreaker` isola fonte com falhas — restante do pipeline continua
- Proteção consistente em todos os adapters sem código duplicado
- `get_rate_limit_stats()` para observabilidade gratuita

### Negative

- spider-nix como dependência de build (flake input + pyproject.toml)

### Risks

| Risk | Probability | Impact | Mitigation |
|------|-------------|--------|------------|
| spider-nix API break | low | medium | Versão pinada no pyproject.toml |
| MRO incorreto em novo adapter | low | low | Documentado no docstring do mixin |
