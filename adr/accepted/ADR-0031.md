## Context

Cerebro v2.0.0 is an Enterprise Knowledge Extraction Platform — a Python CLI/TUI/Dashboard tool that analyzes codebases, runs RAG pipelines, and produces repository intelligence. It currently uses ChromaDB (SQLite) for vector storage, manual Dockerfiles for builds, and Python scripts for ingestion.

The enterprise migration requires replacing these components with production-grade equivalents:

| Component | Current | Enterprise (Google Way) | Rationale |
|-----------|---------|------------------------|-----------|
| **Vector Store** | ChromaDB (SQLite) | Vertex AI Vector Search | Serverless, scales to billions of vectors, <10ms latency |
| **Build System** | Dockerfile manual | Nix `dockerTools.buildImage` | Bit-for-bit reproducible distroless images without Dockerfile |
| **Ingestion** | Python scripts | Cloud Run Jobs | Perfect for batch ETL <10TB. Dataflow is overkill at this scale |
| **Compute/eBPF** | Local / Cloud Run | GKE Standard (Dataplane V2) | Only place to load custom BPF programs with native Cilium |

The critical question: **do all four components belong in the same project?**

### The Boundary Test

A project should be split when:

1. **Deploy cycles are independent** — You update the CLI without touching the cluster. You scale GKE without redeploying the Python package.
2. **Ownership domains differ** — Code intelligence (analysis, RAG, metrics) vs. platform engineering (Terraform, Kubernetes, networking).
3. **Failure domains are isolated** — A bug in the analyzer doesn't bring down the cluster. A GKE node failure doesn't break the local CLI.

All three conditions are met here.

## Decision

**Split into two projects with a clear boundary:**

### `cerebro/` — Intelligence Tool (this project)

The code intelligence platform. Analyzes codebases, runs RAG pipelines, produces metrics, serves dashboards. Ships as a Python package (`pip install cerebro-cli`) and a Nix flake.

**Stays in Cerebro:**

| Component | Implementation | Why it belongs |
|-----------|---------------|----------------|
| Vertex AI Vector Search | New provider replacing ChromaDB | Core RAG backend — this IS the product |
| Nix `dockerTools.buildImage` | Build pipeline addition | Reproducible builds are Cerebro's identity (Nix-first) |
| FastAPI REST API | Already exists (`dashboard_server.py`) | The public interface for integrations |
| Provider abstractions | `interfaces/llm.py`, `interfaces/vector_store.py` | Pluggable backends — the architecture stays clean |

**Does NOT change:**
- CLI (Typer, 8 command groups, 30+ commands)
- TUI (Textual, 6 screens)
- Dashboard (React 18 + Vite)
- Core analysis engine (Tree-Sitter, AST, security scanner)
- Metrics collector (zero-token repository scanning)
- Intelligence system (SIGINT/HUMINT/OSINT/TECHINT)

### `cerebro-infra/` — Platform Engineering (new project)

The production infrastructure that runs Cerebro at enterprise scale. Provisions and manages cloud resources. Ships as Terraform modules and Kubernetes manifests.

| Component | Implementation | Why it's separate |
|-----------|---------------|-------------------|
| GKE Standard (Dataplane V2) | Terraform + Cilium policies | Cluster lifecycle ≠ application lifecycle |
| Cloud Run Jobs (ingestion) | Terraform + Job definitions | Orchestration layer, not business logic |
| eBPF programs | Custom BPF, loaded via Cilium | Kernel-level networking, completely different domain |
| VPC / IAM / monitoring | Terraform modules | Infrastructure provisioning |
| Container registry | Artifact Registry + Nix builds | Image pipeline management |

### The Interface Contract

The two projects communicate through:

```
cerebro (produces)                    cerebro-infra (consumes)
─────────────────                     ────────────────────────
Container image (Nix-built)    →      GKE Deployment / Cloud Run Job
OpenAPI spec (FastAPI)         →      Ingress / Load Balancer config
Vertex AI provider config      →      Vertex AI Vector Search index
Health endpoint (/health)      →      GKE liveness/readiness probes
```

**Rule: Cerebro never imports infrastructure code. cerebro-infra never imports analysis code.**

## Rationale

### Drivers

1. **Independent deploy cycles** — Updating the analyzer shouldn't require a Terraform apply. Scaling the cluster shouldn't require a Python release.
2. **Domain clarity** — Code intelligence engineers work on cerebro. Platform engineers work on cerebro-infra. Different skills, different review cycles.
3. **Cerebro works standalone** — The CLI, TUI, and local ChromaDB all work without any cloud infrastructure. cerebro-infra is an *optional production layer*, not a dependency.
4. **The Google Way** — Use the right tool at the right scale. Cloud Run Jobs for batch ETL (not Dataflow). GKE Standard for eBPF (not Autopilot). Vertex AI Vector Search for scale (not self-hosted Chroma).

### Alternatives Considered

#### Option A: Monorepo (everything in `cerebro/`)

- **Pros:** Single repo, single CI, simpler to start
- **Cons:** Terraform changes trigger Python tests. GKE manifests mixed with CLI code. Unclear ownership.
- **Why rejected:** The deploy cycles are fundamentally different. A monorepo creates artificial coupling between a Python tool and Kubernetes infrastructure.

#### Option B: Cerebro-only (defer all infra)

- **Pros:** Minimal effort, no new repo
- **Cons:** Blocks enterprise deployment. No path to GKE/eBPF. Loses the "Google Way" advantage.
- **Why rejected:** Vector Search and Nix build belong in Cerebro today. But without an infra project, there's no home for GKE, Cloud Run Jobs, or Cilium policies.

#### Option C: Split (CHOSEN)

- **Pros:** Clear boundaries, independent deploys, each project stays focused
- **Cons:** Two repos, cross-repo coordination for breaking changes
- **Why chosen:** Matches the natural boundary between application and infrastructure. Both projects can evolve independently.

### Trade-offs

- **Two repos** — More repos to maintain, but each is simpler and more focused
- **Build dependency** — cerebro-infra consumes container images built by cerebro. Mitigated by publishing images to Artifact Registry with semantic versioning.
- **API contract drift** — cerebro's REST API must stay compatible with cerebro-infra's ingress config. Mitigated by OpenAPI spec as the shared contract.

## Consequences

### Positive

- Cerebro stays a focused, lean intelligence tool (61 modules, 14K LoC)
- cerebro-infra can adopt Terraform, Helm, Cilium without polluting the Python codebase
- Each project has its own test suite, CI/CD pipeline, and release cycle
- Local development needs zero cloud infrastructure (ChromaDB fallback stays)
- Enterprise deployment is a separate, opt-in layer

### Negative

- Two CI/CD pipelines to maintain (GitHub Actions + GitLab CI for each)
- Breaking API changes in Cerebro require coordinated updates in cerebro-infra
- Contributors need to know which repo to target for their changes

### Risks

| Risk | Probability | Impact | Mitigation |
|------|-------------|--------|------------|
| API contract drift between repos | Medium | Medium | Shared OpenAPI spec published by cerebro; contract tests in cerebro-infra |
| cerebro-infra becomes unmaintained | Low | Low | Cerebro works fully standalone; infra is optional |
| Unclear which repo owns Cloud Run Jobs logic | Medium | Low | Rule: business logic in cerebro, orchestration in cerebro-infra |

- [ ] Terraform module: GKE Standard cluster with Dataplane V2

- [ ] Health check integration: GKE probes → cerebro `/health` endpoint
