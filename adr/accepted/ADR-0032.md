## Context

O ADR-Ledger evoluiu de um simples registro de decisoes para um sistema de governanca-as-code que alimenta 4 agentes inteligentes (CEREBRO, SPECTRE, PHANTOM, NEUTRON). Porem, a integridade das decisoes depende exclusivamente do Git, que pode ter historico reescrito via force-push. O content hash atual (SHA256[:12]) so cobre context+decision, nao o ADR completo. Nao existe assinatura digital por decisao, e as regras de governanca sao documentacao, nao codigo executavel.

## Decision

Implementar uma camada de **blockchain privado** em 3 tiers incrementais que eleva o ADR-Ledger de "livro razao contabil" para um sistema de integridade criptografica real:

### Tier 1: Cryptographic Foundation
- **Decision Chain**: Cada ADR aceito vira um bloco com hash SHA256 do arquivo completo, encadeado ao bloco anterior. Qualquer alteracao retroativa quebra a cadeia.
- **Multi-Signature Governance**: Assinaturas Ed25519 por decisao, mapeadas a approval matrix (critical=2, major=1, minor=1, patch=1).
- **Chain Verification**: Comando `adr chain verify` valida integridade de toda a cadeia.

### Tier 2: Trust & Proof
- **Merkle Tree**: Arvore binaria sobre content hashes para provas de inclusao O(log n). Root hash = estado do conhecimento organizacional.
- **Smart Governance Contracts**: Engine executavel que transforma governance.yaml em validadores automaticos com audit log.
- **Cryptographic Snapshots**: Snapshots periodicos assinados encadeando chain tip + merkle root + estatisticas.

### Tier 3: Intelligence & Federation
- **Decision Provenance**: Rastreabilidade end-to-end (research -> analysis -> proposal -> review -> acceptance -> implementation -> outcome).
- **Decision Economics**: Quality scoring com metricas compostas (implementation compliance, review depth, risk analysis).
- **Federated Network**: Compartilhamento cross-org com selective disclosure baseado em topics.

### Arquitetura
Todos os modulos em Python 3.13 sob `.chain/`, usando PyNaCl (libsodium) para Ed25519. Chain state em `.chain/chain.json` (append-only). Chaves privadas gitignored. Integracao completa com CLI, schema JSON, governance YAML e Nix flake.

## Consequences

### Positive
- Integridade criptografica real — qualquer adulteracao e detectavel
- Non-repudiation via assinaturas Ed25519 por decisao
- Provas de inclusao eficientes via Merkle tree
- Governanca enforced por codigo, nao apenas documentacao
- Rastreabilidade completa do ciclo de vida da decisao
- Quality scoring permite feedback loops para melhorar decisoes futuras
- Federation habilita compartilhamento seguro entre organizacoes
- Backward compatible — ADRs existentes funcionam sem modificacao

### Negative
- Adiciona complexidade ao workflow de aceitacao
- Requer gerenciamento de chaves Ed25519 por participante
- PyNaCl como nova dependencia no Nix flake
- Chain state (.chain/chain.json) deve ser versionado no Git
- Curva de aprendizado para novos contribuidores
