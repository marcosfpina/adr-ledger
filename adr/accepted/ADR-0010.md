## Context

**PHANTOM v2.0** é um framework ML "living" para document intelligence, RAG pipelines e classificação de documentos. O projeto combina:

- **Python Core**: CortexProcessor, RAG pipeline, sentiment analysis
- **Rust Components**: IntelAgent (GTK4), Cortex Desktop (Tauri)
- **Infrastructure**: Nix flake para reprodutibilidade, CI/CD robusto

### Estado Atual (2026-01-14)

**✅ Pontos Fortes Implementados**:

```
✓ CI/CD GitHub Actions completo (lint, test, build, security scan)
✓ Nix environment reprodutível (Python 3.13 + Rust + deps)
✓ Security scanning (bandit, cargo audit, detect-secrets)
✓ Type safety (Pydantic models, mypy hints)
✓ Pre-commit hooks + conventional commits
✓ Documentação excelente (CONTRIBUTING.md, SECURITY.md, CODE_OF_CONDUCT.md)
✓ Testes funcionando (24 test cases, 100% pass rate recente)
✓ Multi-linguagem (Python + Rust hybrid)
✓ Modular architecture (core, rag, analysis, pipeline, providers, api, cli)
```

**❌ Gaps Críticos para Produção**:

### 🔴 Tier 1: Bloqueadores (Sem isso, produção = inseguro/inviável)

#### 1. API Incompleta & Insegura
```python
# src/phantom/api/app.py - Endpoints com TODOs
@app.post("/extract")
async def extract(request: ExtractRequest):
    # TODO: Implement using CortexProcessor  ❌
    insights = {"themes": [], ...}  # Mock response

@app.post("/rag/query")
async def rag_query(question: str):
    # TODO: Implement RAG query  ❌
```

**Riscos**:
- Endpoints não funcionais em produção
- Sem autenticação (qualquer um pode chamar API)
- Sem rate limiting (vulnerável a DoS)
- Sem validação de input size (OOM attacks)

#### 2. Zero Observabilidade
```
❌ Nenhum metric export (Prometheus, StatsD)
❌ Sem distributed tracing (OpenTelemetry, Jaeger)
❌ Logging não estruturado (print statements, não JSON)
❌ Sem alertas configurados
❌ Sem dashboards (Grafana)
```

**Impacto**: Cego em produção. Quando algo quebra, sem telemetria para debug.

#### 3. Sem Persistência
```
❌ Nenhum banco de dados (dependência de arquivos locais)
❌ Sem migrations (Alembic)
❌ Sem backup strategy
❌ Sem cache distribuído (Redis/Memcached)
```

**Impacto**: Não escalável, sem state management, perda de dados em crash.

#### 4. Containerização Ausente

## Decision

**Adotar roadmap incremental de 4 sprints** para atingir production readiness, priorizando gaps Tier 1 (críticos) primeiro.

### Princípios Norteadores

1. **Security-First**: Autenticação e hardening em Sprint 1
2. **Observability-Driven**: Metrics desde Sprint 1, full stack em Sprint 3
3. **Incremental**: Cada sprint entrega valor isolado
4. **SLO-Oriented**: Definir e medir SLOs desde o início

### Target SLOs (Service Level Objectives)

```yaml
SLOs:
  availability:
    target: 99.9%
    measurement_window: 30d
    error_budget: 43.2min/month

  latency:
    p50: <200ms
    p95: <500ms
    p99: <1000ms
    measurement: /process endpoint

  error_rate:
    target: <0.1%
    excludes: [4xx client errors]

  throughput:
    target: 100 docs/min (single instance)
    scale_trigger: 80% capacity

  data_durability:
    target: 99.999%
    backup_frequency: 1h
    recovery_time: <15min
```

## Roadmap: 4 Sprints para Production

### 📋 Sprint 1: Security Hardening (1.5 semanas)

**Goal**: API segura e auditável

**Tasks**:

```yaml
1. JWT Authentication:
   - Implementar /auth/login, /auth/refresh
   - JWT validation middleware
   - Role-based access control (RBAC)
   - Expiration + refresh token flow

2. Rate Limiting:
   - slowapi integration (Redis-backed)
   - Limites: 100 req/min por IP, 1000 req/hour por user
   - Custom limits por endpoint (/process: 10 req/min)

3. Secrets Management:
   - GCP Secret Manager integration
   - Nunca hardcode API keys
   - Rotation automática (90 dias)

4. Structured Logging:
   - structlog + JSON formatter
   - Log levels: DEBUG, INFO, WARNING, ERROR, CRITICAL
   - Context: request_id, user_id, timestamp, stack_trace
   - Redact sensitive fields (API keys, passwords)

5. Input Validation:
   - File size limits (max 50MB por upload)
   - Content-Type validation
   - Path traversal protection
   - SQL injection safeguards (prepared statements)
```

**Deliverables**:
- ✅ `/auth/*` endpoints funcionais
- ✅ Rate limiting ativo em todos os endpoints
- ✅ Logs estruturados em JSON
- ✅ Security audit report
- ✅ Pen-test inicial (automated OWASP ZAP)

**Success Metrics**:
- Zero hardcoded secrets em código
- 100% de endpoints com rate limiting
- Logs parseáveis por log aggregators

---

### 📋 Sprint 2: Data Layer (1.5 semanas)

**Goal**: Persistência robusta e escalável

**Tasks**:

```yaml
1. PostgreSQL Setup:
   - Schema design:
     * users (id, email, hashed_password, role, created_at)
     * documents (id, user_id, filename, status, created_at)
     * insights (id, document_id, themes, patterns, embeddings)
     * audit_logs (id, user_id, action, timestamp, metadata)
   - Connection pooling (asyncpg, pool_size=20)
   - Indices: user_id, document_id, created_at

2. Alembic Migrations:
   - Initial schema (0001_initial.py)
   - Migration automation em CI/CD
   - Rollback procedures

3. Redis Cache:
   - Cache layer para embeddings (TTL: 1h)
   - Session storage para auth tokens
   - Rate limiting counters
   - Connection pooling

4. Health Checks:
   - /health/live (processo ativo?)
   - /health/ready (DB + Redis conectados?)
   - /health/startup (migrations aplicadas?)
   - Kubernetes-compatible

5. Backup Strategy:
   - PostgreSQL: WAL archiving + PITR (Point-In-Time Recovery)
   - Redis: RDB snapshots a cada 1h
   - Retenção: 7 dias local, 30 dias GCS
```

**Deliverables**:
- ✅ PostgreSQL schema completo
- ✅ Alembic migrations funcionais
- ✅ Redis cache operacional
- ✅ Health checks /health/* prontos
- ✅ Backup automated scripts

**Success Metrics**:
- Migrations aplicam em <30s
- Cache hit rate >70% (embeddings)
- Health checks respondem em <100ms

### 📋 Sprint 3: Observability (1 semana)

**Goal**: Visibilidade total do sistema em produção

**Tasks**:

```yaml
1. Prometheus Metrics:
   - prometheus-fastapi-instrumentator
   - Custom metrics:
     * phantom_documents_processed_total (counter)
     * phantom_processing_duration_seconds (histogram)
     * phantom_llm_tokens_used_total (counter)
     * phantom_cache_hit_rate (gauge)
     * phantom_db_connections_active (gauge)
   - /metrics endpoint

2. Grafana Dashboards:
   - Dashboard 1: SLO Overview (uptime, latency, errors)
   - Dashboard 2: Business Metrics (docs/hour, costs, users)
   - Dashboard 3: Infrastructure (CPU, RAM, GPU, disk)
   - Dashboard 4: Errors & Debugging (error rate, stack traces)

3. Alerting:
   - AlertManager + PagerDuty integration
   - Alerts:
     * API down >1min (critical)
     * Error rate >1% (warning)
     * P95 latency >500ms (warning)
     * Disk usage >80% (warning)
     * VRAM usage >90% (critical)
   - On-call rotation

4. Load Testing:
   - Locust scenarios:
     * Scenario 1: 100 concurrent users, /process endpoint
     * Scenario 2: 1000 req/min, mixed endpoints
     * Scenario 3: Spike test (0→500 users em 10s)
   - Identify bottlenecks, optimize

5. Distributed Tracing (Optional):
   - OpenTelemetry + Jaeger (se tempo permitir)
   - Trace: API request → LLM call → DB query → response
```

**Deliverables**:
- ✅ Prometheus metrics exportando
- ✅ 4 Grafana dashboards operacionais
- ✅ Alertas configurados e testados
- ✅ Load test report (bottlenecks identificados)
- ✅ Performance tuning (P95 <500ms)

**Success Metrics**:
- Alert false-positive rate <5%
- Load test: 100 docs/min sustained
- P95 latency <500ms under load

**Goal**: Deployment reproduzível e escalável

```yaml
1. Dockerfile Multi-Stage:

   - Stage 2: Runtime (minimal image, <500MB)
   - Otimizações: layer caching, .dockerignore
   - Security scan: Trivy (zero CRITICAL vulns)

2. Docker Compose (Dev/Staging):
   - Services: phantom-api, postgresql, redis, prometheus, grafana

3. Kubernetes Manifests:

4. CI/CD Pipeline:
   - GitHub Actions workflow:
     * Build → Test → Security Scan
     * Build Docker image → Push to GCR
     * Deploy to GKE (staging)
     * Smoke tests
     * Manual approval gate
     * Deploy to GKE (production)
     * Post-deploy verification

5. Production Readiness Review:

     [ ] Runbooks for common incidents
```

## Risk Mitigation

### Technical Risks

| Risk | Mitigation |
|------|------------|
| **PostgreSQL performance bottleneck** | Connection pooling (asyncpg), read replicas, query optimization |
| **Redis OOM (out of memory)** | TTL policies, LRU eviction, memory limits, monitoring |
| **LLM provider rate limits** | Exponential backoff, circuit breaker, fallback providers |
| **GPU OOM during inference** | VRAM monitoring (existing), batch size tuning, model quantization |
| **K8s cluster instability** | Multi-zone deployment, pod disruption budgets, health checks |

### Operational Risks

| Risk | Mitigation |
|------|------------|
| **On-call burnout** | Automated runbooks, good alerting (low false positives), SLO-based escalation |
| **Deployment failures** | Smoke tests, canary deployments (10% → 50% → 100%), instant rollback |
| **Data loss** | PITR backups, tested restore procedures, geo-redundant storage |
| **Cost overruns** | Budget alerts, auto-scaling limits, cost dashboards, monthly reviews |
