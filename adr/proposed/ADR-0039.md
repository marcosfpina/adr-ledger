## Context

During Phase 3 validation of E2E trace propagation (Task #41, ROADMAP), five anomalies were detected that require deeper investigation. These are documented here as a structured investigation backlog to prevent knowledge loss and ensure systematic follow-up.

All anomalies were discovered on 2026-02-15 while validating OpenTelemetry OTLP export from `spectre-proxy` to Jaeger.

---

## ANO-01: tonic gRPC Batch Exporter Silent Failure

**Severity**: HIGH
**Category**: Reliability
**Status**: Workaround applied (ADR-0038), root cause unknown

### Observed Behavior

The `opentelemetry-otlp` gRPC/tonic exporter (`new_exporter().tonic()`) initialized without error, the batch processor ran normally, but **zero traces reached Jaeger**. No errors were logged anywhere — not in application logs, not in stderr, not via the OpenTelemetry global error handler.

### Evidence

```
# Jaeger OTLP gRPC endpoint (4317) is listening and accepts connections
# Jaeger OTLP HTTP endpoint (4318) works perfectly with manual curl
# tracing-opentelemetry layer creates spans (confirmed via tower_http debug logs)
# Batch exporter runs on schedule (no timeout, no backpressure)
# Zero spans exported over gRPC after 60+ requests and 30+ seconds
```

### Hypotheses to Investigate

1. **tonic 0.9.2 vs workspace tonic 0.11 conflict** — Two tonic versions coexist. The opentelemetry-otlp crate uses 0.9 while the workspace depends on 0.11. Could cause runtime type confusion in gRPC channel setup.
2. **HTTP/2 handshake failure** — tonic 0.9 may use an h2 version incompatible with Jaeger's gRPC server, causing silent connection drops.
3. **Protobuf schema mismatch** — `opentelemetry-proto 0.4` (via prost) may generate protobuf messages that Jaeger's OTLP gRPC receiver silently rejects.
4. **Tokio runtime context loss** — The batch exporter spawns tasks on `opentelemetry_sdk::runtime::Tokio`. If the runtime handle is lost or mismatched, export tasks may silently abort.
5. **TLS negotiation on plaintext port** — tonic may attempt TLS by default on `http://localhost:4317`, causing a silent handshake failure.

### Investigation Plan

```bash
# 1. Enable tonic debug logging
RUST_LOG=info,tonic=debug,h2=debug,hyper=debug cargo run -p spectre-proxy

# 2. Capture gRPC traffic
tcpdump -i lo -w /tmp/grpc-capture.pcap port 4317

# 3. Test with grpcurl
grpcurl -plaintext localhost:4317 list

# 4. Isolate tonic version
# Create minimal repro with only tonic 0.9.2 (no workspace 0.11)

# 5. Test with OTEL collector instead of Jaeger direct
docker run otel/opentelemetry-collector --config otel-config.yaml
```

### Workaround

Switched to HTTP/protobuf exporter (ADR-0038). Works immediately and reliably.

## ANO-02: OTEL_SERVICE_NAME Environment Variable Ignored

**Severity**: MEDIUM
**Category**: Configuration / Standards Compliance

### Observed Behavior

Setting `OTEL_SERVICE_NAME=spectre-proxy-v3` had no effect on the service name in Jaeger. All traces appeared under `spectre-proxy` regardless.

### Root Cause (Known)

The `spectre_observability::init()` function receives the service name as a hardcoded parameter:

```rust
// crates/spectre-proxy/src/main.rs
spectre_observability::init("spectre-proxy")?;

// crates/spectre-observability/src/lib.rs
KeyValue::new(resource::SERVICE_NAME, service_name.to_string()),
```

The standard `OTEL_SERVICE_NAME` env var is never read.

### Impact

- **Non-compliant** with [OpenTelemetry SDK specification](https://opentelemetry.io/docs/specs/otel/configuration/sdk-environment-variables/) which mandates `OTEL_SERVICE_NAME` support
- Cannot override service name in different deployment environments (staging, canary, blue/green) without code changes
- Confuses operators who expect standard OTEL env vars to work

### Fix (Straightforward)

```rust
pub fn init(service_name: &str) -> Result<()> {
    let effective_name = std::env::var("OTEL_SERVICE_NAME")
        .unwrap_or_else(|_| service_name.to_string());
    // ... use effective_name for Resource
}
```

## ANO-03: Batch Exporter Flush Latency (15-25 seconds)

**Severity**: MEDIUM
**Category**: Observability / Operational

### Observed Behavior

After sending HTTP requests, traces took **15-25 seconds** to appear in Jaeger. In some cases, traces from a batch of 20+ requests didn't appear after 10 seconds, leading to false conclusions about export failure.

### Expected Behavior

The `opentelemetry_sdk` batch exporter default schedule interval is **5 seconds** (`BatchConfig::default().scheduled_delay`). Traces should appear within ~5-7 seconds.

### Hypotheses

1. **Batch size not reached** — Default `max_export_batch_size` is 512. With low traffic, the exporter waits for the schedule interval, but the actual delay observed (15-25s) exceeds the 5s default.
2. **HTTP/protobuf reqwest connection pool warmup** — First export may incur DNS resolution + TCP handshake + TLS (if applicable), adding latency to the first batch.
3. **Jaeger indexing delay** — Jaeger's in-memory storage may take time to index traces for API queries, separate from OTLP ingestion.
4. **Multiple batch intervals** — If the batch exporter encounters transient errors, it may retry on the next interval, accumulating delay.

### Investigation Plan

```bash
# 1. Tune batch config (in code)
.install_batch(opentelemetry_sdk::runtime::Tokio)
# Add: .with_batch_config(BatchConfig::default().with_scheduled_delay(Duration::from_secs(2)))

# 2. Test with force flush
# Add endpoint that calls global::force_flush() and measure Jaeger latency

# 3. Instrument the exporter
RUST_LOG=opentelemetry_sdk::trace::span_processor=debug
```

## ANO-04: tonic Version Conflict (0.9.2 vs 0.11)

**Severity**: HIGH
**Category**: Dependency Management / Build Integrity

### Observed Behavior

Two incompatible versions of `tonic` coexist in `Cargo.lock`:

```
tonic 0.9.2  — pulled by opentelemetry-otlp 0.14
tonic 0.11.0 — workspace dependency for spectre-proxy gRPC
```

These pull different versions of `prost`, `h2`, `hyper`, and `http` crates, creating a diamond dependency.

### Impact

- **Binary size inflation** — Two copies of gRPC stack compiled into binary
- **Potential runtime confusion** — Types from tonic 0.9 and 0.11 are incompatible at the type system level, but could share runtime resources (connection pools, TLS contexts) in unexpected ways
- **Likely root cause of ANO-01** — The silent gRPC export failure may stem from this conflict

### Resolution Options

| Option | Effort | Risk |
|--------|--------|------|
| Upgrade opentelemetry-otlp to 0.27+ (uses tonic 0.12+) | High — breaking API changes across opentelemetry ecosystem | Medium |
| Pin workspace tonic to 0.9 | Low — but blocks other gRPC use cases | Low |
| Remove tonic from workspace, use only via opentelemetry | Medium — refactor any direct gRPC code | Low |
| Use HTTP-only OTLP (current workaround) | Already done | None |

### Recommendation

Defer until opentelemetry-rust stable 1.0 release (expected Q2 2026), which will align all transitive dependencies. Continue with HTTP/protobuf workaround.

## ANO-05: Phantom WebSocket Traffic on Port 3000

**Severity**: LOW-MEDIUM
**Category**: Security / Environment

### Observed Behavior

Jaeger captured continuous traces for `GET /ws/socket.io/?EIO=4&transport=websocket` on port 3000. This traffic appeared **immediately** after starting the proxy, constituting ~60-80% of all captured traces.

### Evidence

```
92 total traces in 3-minute window:
  /health:    42
  websocket:  49  ← unexplained
  other:       5
```

The `spectre-proxy` has no WebSocket endpoint configured. These requests return 401 (Unauthorized) but still generate trace spans.

### Hypotheses

1. **Open WebUI residual** — Port 3000 was previously used by Open WebUI. Browser tabs or service workers may still attempt WebSocket connections to `localhost:3000`.
2. **Browser extension** — A browser extension (e.g., dev tools, live reload) may poll `localhost:3000` for WebSocket connections.
3. **systemd service auto-restart** — Open WebUI may have a systemd service that attempts reconnection to port 3000.
4. **Malicious probing** — Unlikely in dev environment, but should be ruled out.

### Investigation Plan

```bash
# 1. Identify the source
tcpdump -i lo port 3000 -nn | grep "socket.io"

# 2. Check browser connections
# Open browser dev tools → Network tab → filter WS

# 3. Check systemd services
systemctl list-units | grep -i web
systemctl list-units | grep -i open

# 4. Check if traffic stops when browser is closed
pkill -f firefox; pkill -f chromium
# Then monitor: curl localhost:16686/api/traces?service=spectre-proxy&lookback=1m
```

### Immediate Action

Not blocking. Traces are noisy but harmless. Consider adding a path filter to the TraceLayer to exclude `/ws/` paths from tracing if noise becomes problematic.
