## Contexto

A preparação do branch orphan `og` como ponto de partida open source exige uma
garantia auditável de que nenhum secret, credencial ou dado sensível está presente
no working tree publicado. Isso levanta a questão de qual ferramenta de secret
scanning adotar no pipeline de CI.

O candidato natural no ecossistema seria o **Trivy** (Aqua Security), amplamente
adotado e com suporte nativo a GitHub Actions. No entanto, o Trivy esteve envolvido
em um **incidente grave de segurança no GitHub**, o que torna sua adoção incompatível
com os princípios de supply chain security que este projeto documenta em seus próprios ADRs.

## Decisão

Adotar o **Gitleaks** como ferramenta exclusiva de secret scanning neste repositório.
Rejeitar explicitamente o Trivy para qualquer papel de segurança neste pipeline.

### Componentes entregues

- **`.gitleaks.toml`** — Configuração com allowlist para falsos positivos conhecidos:
  `.chain/keys/*.pub` (public keys Ed25519 legítimas), `flake.lock`, hashes SHA-256
  da chain e chaves públicas no formato hex de 128 chars
- **`.github/workflows/secret-scan.yml`** — Workflow dedicado usando `gitleaks-action@v2`,
  sem dependências Nix, ativado em push para `main`/`og`, PRs e `workflow_dispatch`
- **`flake.nix` devShell** — `pkgs.gitleaks` adicionado para scan local via `nix develop`
- **`.gitignore` endurecido** — `.env*`, `*.pem`, `*.p12`, `*.pfx`, `*.jks`, `.mcp.json`

## Rationale

### Por que não Trivy

O Trivy é um scanner multi-propósito (CVEs, containers, IaC, secrets). Essa amplitude
é sua principal fraqueza no contexto desta decisão:

1. **Incidente de segurança**: O Trivy esteve envolvido em um incidente grave no GitHub.
   Uma ferramenta de segurança que demonstra falhas na sua própria cadeia de fornecimento
   não pode ser considerada um guardião confiável para outros sistemas. O princípio é
   simples: **não se usa um alarme quebrado para proteger uma casa**.

2. **Escopo desnecessário**: Secret scanning é uma feature secundária do Trivy. Adotar
   uma ferramenta multi-propósito quando o objetivo é específico expande a superfície de
   ataque sem benefício proporcional.

3. **Exclusão explícita e documentada**: Esta ADR serve como registro formal para prevenir
   que contribuidores futuros adotem o Trivy por conveniência ou familiaridade sem revisar
   o histórico desta decisão.

### Por que Gitleaks

- **Purpose-built**: Criado exclusivamente para detecção de secrets e credenciais
- **Superfície de ataque mínima**: Faz uma coisa, faz bem
- **Integração nativa**: `gitleaks-action@v2` funciona diretamente no GitHub Actions
- **Configurável**: `.gitleaks.toml` permite allowlists precisas para falsos positivos
- **Disponível no Nix**: `pkgs.gitleaks` sem friction de ambiente
- **Resultado verificado**: `no leaks found` em scan de 1.28 MB do branch `og`

## Consequências

### Positivas

- Branch `og` escaneado e auditado antes de publicação pública
- CI roda automaticamente em cada push para `main` e `og`
- Precedente documentado: qualquer mudança para outro scanner exige nova ADR
- Scan local disponível sem sair do ambiente Nix: `nix develop --command gitleaks detect --source . --config .gitleaks.toml --verbose --no-git`

### Negativas

- CVE scanning e análise de vulnerabilidades de dependências não estão cobertos por
  esta decisão — requerem ADR separada quando necessário
- `.gitleaks.toml` precisa ser mantido à medida que novos padrões de dados internos
  forem introduzidos
