## Context

A stack inteligente (CEREBRO, PHANTOM, NEUTRON) está crescendo em complexidade:

- **CEREBRO**: Knowledge extraction, RAG queries, GCP credit management
- **PHANTOM**: Document intelligence, semantic chunking, LLM classification
- **NEUTRON**: ML orchestration, adaptive pipelines, infrastructure management
- **Domain services**: ai-agent-os (system monitoring), securellm-bridge (LLM proxy), ml-offload-api (inference), ragtex (RAG), arch-analyzer (NixOS analysis)

### Problema: Tight Coupling com REST APIs

A abordagem tradicional seria REST APIs diretas:

```
CEREBRO --HTTP--> PHANTOM --HTTP--> ml-offload-api
   |                |
   |                +--HTTP--> ragtex
   |
   +--HTTP--> NEUTRON --HTTP--> securellm-bridge
```

**Problemas observados**:

1. **Tight Coupling**: CEREBRO precisa conhecer endpoints de PHANTOM, ragtex, ml-offload-api
2. **Cascading Failures**: Se ml-offload-api cai, PHANTOM falha, CEREBRO falha
3. **Manual Retry Logic**: Cada serviço implementa próprio retry, circuit breaker, timeout
4. **Zero Observability**: Sem visibilidade centralizada de requests cross-service
5. **Scaling Complexity**: Escalar PHANTOM requer atualizar load balancer, DNS, service discovery

### Exemplo Concreto: RAG Query Workflow

**Workflow atual (REST, tight coupling)**:

1. User: `cerebro rag query "How does auth work?"`
2. CEREBRO → HTTP POST `/phantom/analyze` (bloqueante)
3. PHANTOM → HTTP POST `/ml-offload-api/inference` (bloqueante)
4. ml-offload-api → processa → retorna response
5. PHANTOM → processa → retorna response
6. CEREBRO → retorna ao user

**Problemas**:
- Se ml-offload-api timeout (30s), PHANTOM timeout, CEREBRO timeout
- User aguarda 30s sem feedback
- Retry manual em cada camada
- Logs fragmentados (3 serviços, 3 formatos diferentes)

## Decision

Adotamos **Event-Driven Architecture (EDA)** usando **NATS JetStream** como message broker central.

### Event Schema Example

```rust
// Event structure (spectre-events crate)
pub struct Event {
    pub id: EventId,              // UUID for deduplication
    pub event_type: EventType,    // Enum: LlmRequest, RagQuery, etc
    pub timestamp: DateTime<Utc>, // ISO 8601
    pub service_id: ServiceId,    // Publisher identifier
    pub correlation_id: CorrelationId, // Trace across services
    pub payload: serde_json::Value,    // Type-specific data
}

// Example: RAG query event
EventType::RagQuery {
  payload: {
    "query": "How does auth work?",
    "top_k": 5,
    "filters": {"project": "CEREBRO"}
  }
}
```

### Communication Patterns

#### 1. Pub/Sub (1-to-many)

```rust
// Publisher: PHANTOM publishes document indexed
let event = Event::new(
    EventType::DocumentIndexed,
    service_id,
    json!({"document_id": "doc-123", "chunks": 42})
);
bus.publish(&event).await?;

// Subscribers: CEREBRO, SPECTRE observability
// Both receive the same event simultaneously
```

#### 2. Request-Reply (Synchronous RPC)

```rust
// CEREBRO requests LLM inference
let request = Event::new(EventType::LlmRequest, ...);
let response = bus.request(&request, timeout).await?;
// Blocks until reply or timeout
```

#### 3. Queue Groups (Load Balancing)

```rust
// 3 ml-offload-api workers subscribe to same subject
let sub = bus.subscribe_queue("inference.request.v1", "ml-workers").await?;
// NATS delivers each message to ONE worker (round-robin)
```

### Event Versioning Strategy

Events use semantic versioning in subject names:

- `llm.request.v1` - Initial version
- `llm.request.v2` - Breaking change (new required field)
- Both versions coexist during migration period
- Subscribers can subscribe to both: `llm.request.*`

## Rationale

### Por que Event-Driven?

| Propriedade          | REST APIs (Sync)        | Events (Async)                |
|----------------------|-------------------------|-------------------------------|
| **Coupling**         | Tight (knows endpoints) | Loose (knows event types)     |
| **Failures**         | Cascading               | Isolated (queue buffers)      |
| **Retry Logic**      | Manual per service      | Built-in (NATS redelivery)    |
| **Observability**    | Manual instrumentation  | Automatic (all events logged) |
| **Scaling**          | Coordinated             | Independent (queue depth)     |
| **Throughput**       | ~10k req/sec            | 1M+ msgs/sec (NATS)           |
| **Backpressure**     | HTTP 429, manual        | Queue depth, automatic        |

### Por que NATS (vs Kafka, RabbitMQ)?

**NATS Advantages**:
1. **Simplicity**: Single binary, zero dependencies (no ZooKeeper)
2. **Performance**: 11M+ msgs/sec (core NATS), 1M+ with JetStream persistence
3. **Lightweight**: ~15MB binary, <100MB RAM baseline
4. **Built-in Patterns**: Pub/sub, request-reply, queue groups natively
5. **Cloud-Native**: Designed for Kubernetes, Docker, microservices
6. **Security**: TLS, auth, encryption first-class

**JetStream Benefits** (over core NATS):
1. **Persistence**: Messages survive broker restart (7-day retention)
2. **At-Least-Once**: Guaranteed delivery with ACKs
3. **Replay**: Consumers can replay historical events
4. **Stream Replication**: Multi-node redundancy

### SPECTRE Crates Architecture

**Phase 0 (Complete)**:
- `spectre-core`: Foundation types (ServiceId, CorrelationId, Config, Error)
- `spectre-events`: NATS client, Event struct, 30+ EventTypes, Publisher/Subscriber traits

**Phase 1 (In Progress)**:
- `spectre-proxy`: HTTP/gRPC gateway, TLS termination, auth, rate limiting
- `spectre-secrets`: AES-GCM encryption, Argon2 key derivation, rotation policies

**Phase 2 (Planned)**:
- `spectre-observability`: Wildcard subscriber, TimescaleDB ingestion, Neo4j graphs, ML anomaly detection

## Consequences

### Workflow Transformation: RAG Query (Event-Driven)

**Before (REST)**:
```
User → CEREBRO --HTTP--> PHANTOM --HTTP--> ml-offload-api
Latency: 30s (synchronous blocking)
Failure: Cascading (any service down = total failure)
Observability: Fragmented logs across 3 services
```

**After (Events)**:
```
User → CEREBRO --publish--> rag.query.v1 (NATS)
                               ↓
                          PHANTOM (subscribes)
                               ↓ --publish--> inference.request.v1
                          ml-offload-api (subscribes)
                               ↓ --publish--> inference.response.v1
                          PHANTOM (receives)
                               ↓ --publish--> rag.response.v1
                          CEREBRO (receives)
                               ↓
User ← CEREBRO (returns result)

Latency: ~500ms (async, non-blocking where possible)
Failure: Isolated (NATS buffers events if service down, retry automatic)
Observability: ALL events in TimescaleDB (single query shows full trace)
```

### Observability Example

```sql
-- TimescaleDB: Find all events in a trace
SELECT
  event_type,
  service_id,
  timestamp,
  payload

WHERE correlation_id = 'trace-abc123'
ORDER BY timestamp;

-- Result: Complete trace of RAG query workflow
-- rag.query.v1         (CEREBRO)     2026-01-11 14:23:45
-- inference.request.v1 (PHANTOM)     2026-01-11 14:23:45.120
-- inference.response.v1 (ml-offload) 2026-01-11 14:23:45.890
-- rag.response.v1      (CEREBRO)     2026-01-11 14:23:46.020
-- Total: 1.02s end-to-end
```

### Fault Tolerance Example

**Scenario**: ml-offload-api crashes during high load

**Event-Driven Behavior**:
1. PHANTOM publishes `inference.request.v1` → NATS queue
2. ml-offload-api down → events accumulate in JetStream (persistent)
3. NATS monitoring: queue depth alert (>1000 msgs)
4. Auto-scaling (NEUTRON): Spawns 2 additional ml-offload-api replicas
5. Replicas join queue group "ml-workers"
6. NATS drains queue to replicas (load balanced)
7. No messages lost, requests eventually processed

**REST Behavior (comparison)**:
1. PHANTOM HTTP POST → timeout (30s)
2. PHANTOM retry logic (3 attempts × 30s = 90s)
3. PHANTOM gives up, returns 503 to CEREBRO
4. CEREBRO propagates error to user
5. User retries manually
6. Data lost (no persistence)

### Trade-offs Accepted

1. **Eventual Consistency**: RAG query may see slightly stale data if PHANTOM still processing index updates
   - **Mitigation**: Accept 1-5s staleness, use event timestamps for debugging

2. **Debugging Complexity**: Distributed traces harder than stack traces
   - **Mitigation**: Correlation IDs, TimescaleDB queries, Neo4j graph visualization

3. **NATS as SPOF**: If NATS down, fleet degraded
   - **Mitigation**: NATS clustering (3 nodes), JetStream replication, circuit breakers for degraded mode (local caching)
