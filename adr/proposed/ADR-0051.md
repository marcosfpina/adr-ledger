## Context

The Phantom Cortex Desktop GUI connects to a local llamacpp server (port 8081) for LLM inference. On the development workstation (RTX 3050 6GB Laptop GPU), inference runs at **0.69 tokens/second** — effectively unusable for interactive chat.

Root cause analysis revealed **VRAM contention**: the llamacpp server holds only 98 MiB of VRAM while an unrelated Python process (`python -m src.server`, PID 3080, likely from another monorepo service) consumes ~2GB. With the 8B Q4_K_S model requiring ~4.5GB and only 6GB total available, the GPU is at 94% VRAM / 100% utilization with constant memory pressure.

There is currently **no VRAM arbitration** between services in the monorepo. Any process can claim GPU memory, and there are no startup checks, budgets, or priorities.

### Observed metrics (2026-03-25)

| Metric | Value |
|--------|-------|
| GPU | RTX 3050 6GB Laptop |
| VRAM total | 6144 MiB |
| VRAM used | 5804 MiB (94%) |
| GPU utilization | 100% |
| llamacpp VRAM | 98 MiB |
| Competing Python process | 1988 MiB |
| Model | L3-8B-Stheno-v3.2-Q4_K_S |
| Inference speed | 0.69 tok/s generation, 9.18 tok/s prompt |
| Expected speed (uncontended) | ~8-15 tok/s generation |

## Decision

Adopt a **VRAM budget strategy** with the following rules:

1. **Embedding workloads (sentence-transformers) MUST run on CPU.** The `EmbeddingGenerator` already defaults to `device="cpu"` — this must be enforced, not optional.

2. **llamacpp gets GPU priority.** On startup, `cortex_api` and `phantom-api` must check available VRAM and warn/block if insufficient for the configured model.

3. **VRAM pre-flight check.** Before starting llamacpp or any GPU-intensive service, query `nvidia-smi` for free VRAM and compare against the model's estimated requirement. Emit a clear warning or auto-adjust `--ngl` (number of GPU layers) to fit.

4. **Hardware-tier budget table.** Define expected allocations per GPU class:

| GPU Tier | Total VRAM | llamacpp budget | Max model size | Recommended |
|----------|-----------|-----------------|----------------|-------------|
| 6 GB | 6144 MiB | 5500 MiB | 8B Q4 (tight) | 3B-4B Q4 |
| 8 GB | 8192 MiB | 7500 MiB | 8B Q4 (comfortable) | 8B Q4 |
| 12 GB | 12288 MiB | 11000 MiB | 13B Q4 | 8B-13B Q4 |
| 24 GB | 24576 MiB | 22000 MiB | 30B Q4 | 30B Q4/Q5 |

5. **No other service may load GPU models without explicit opt-in.** Services in the monorepo that use PyTorch/transformers must default to CPU unless `PHANTOM_GPU_ALLOW=1` or equivalent is set.

## Rationale

### Drivers

- Interactive chat in Cortex Desktop is the primary user-facing feature and must be responsive
- 6GB GPUs are the minimum viable hardware target — VRAM is the scarcest resource
- Multiple services in the monorepo can independently claim GPU memory without coordination

### Alternatives Considered

#### Option A: Force smaller models only
- **Pros:** Simple, guaranteed to fit
- **Cons:** Degrades output quality, doesn't solve the coordination problem
- **Why rejected:** Treats the symptom, not the cause

#### Option B: GPU time-slicing (MPS/MIG)
- **Pros:** True multi-tenant GPU sharing
- **Cons:** Requires Volta+ for MIG, adds complexity, not available on consumer GPUs
- **Why rejected:** RTX 3050 doesn't support MIG; MPS adds latency

#### Option C: Single GPU owner (llamacpp only)
- **Pros:** Maximum VRAM for inference, simple
- **Cons:** Other services can't use GPU even when llamacpp is idle
- **Why rejected:** Too restrictive, but adopted as the default stance with opt-in exceptions

### Trade-offs

- **Embedding speed**: CPU embeddings are ~2-5x slower than GPU, but embedding is a batch operation (indexing time), not interactive latency. Acceptable.
- **Flexibility vs safety**: Locking GPU to llamacpp prevents other legitimate GPU workloads. Mitigated by opt-in flag.

## Consequences

### Positive

- Predictable inference latency (target: 8-15 tok/s on 6GB for 8B Q4)
- Clear ownership of GPU resources
- Startup diagnostics prevent silent performance degradation
- Hardware-tier table guides model selection

### Negative

- Embedding indexing slightly slower (CPU vs GPU)
- Requires discipline across monorepo services
- Pre-flight check adds startup latency (~1s)

### Risks

| Risk | Probability | Impact | Mitigation |
|------|-------------|--------|------------|
| Rogue process loads GPU model | High | High | VRAM guard + `PHANTOM_GPU_ALLOW` env gate |
| Model doesn't fit after VRAM check passes | Low | Medium | Conservative budget (leave 500MiB headroom) |
| User overrides CPU-only for embeddings | Low | Medium | Log warning when `device=cuda` is used |
