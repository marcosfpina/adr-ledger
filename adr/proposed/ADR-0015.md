## Context

### Problema: Code Smells Críticos

A análise do codebase revelou **code smells críticos** que comprometem maintainability, compliance e portabilidade:

#### 1. Hardcoded Paths (5+ instâncias)

**Phantom** (`src/phantom/neutron/sentinel_integration.py:24-26`):
```python
neutron_path = Path("/home/kernelcore/arch/neutron")
sys.path.insert(0, str(neutron_path))
```

**Problemas**:
- Não funciona em ambientes diferentes (Docker, outros usuários, CI/CD)
- sys.path manipulation é anti-pattern (implicit dependencies)
- Quebra em deployment sem estrutura exata de diretórios

#### 2. Duplicação Massiva de Código

| Componente | Localizações | LOC Total | Impacto |
|------------|-------------|-----------|---------|
| **SENTINEL Core** | Neutron (302 LOC) + Phantom fallback (92 LOC) | **727 LOC** | Compliance logic duplicado |
| **CORTEX Chunking** | Phantom + Cerebro | **~200 LOC** | Semantic chunking duplicado |
| **CEREBRO in Phantom** | Phantom duplica Cerebro | **471 LOC** | RAG logic completo duplicado |
| **Hardcoded configs** | 3+ projetos | **~50 LOC** | Configuration boilerplate |
| **TOTAL** | - | **~1,448 LOC** | - |

**Consequências**:
- Bug fix em SENTINEL requer mudanças em 3 arquivos
- Inconsistências: Phantom tem versão desatualizada do SENTINEL
- Test coverage fragmentado (testes duplicados)

#### 3. Compliance Opcional (Risco Crítico)

**Phantom** (`sentinel_integration.py:41-92`):
```python
try:
    from neutron.compliance.sentinel import ComplianceGuardrail
except ImportError:
    # Fallback mode - BYPASS SENTINEL
    class ComplianceGuardrail:
        def __init__(self, *args, **kwargs):
            self.bypass_mode = True

        def validate(self, text: str) -> ValidationResult:
            # NO VALIDATION - LGPD RISK
            return ValidationResult(approved=True, violations=[])
```

**Problema**: Se Neutron não está disponível, Phantom bypassa compliance completamente.

**Risco**: PII (emails, phones, SSNs) pode ser processado sem validação → **Violação LGPD Art. 46**.

#### 4. Dependências Implícitas

**Dependency Graph Atual**:
```
Phantom
  ├─> Neutron (via sys.path injection) ❌ Não declarado em pyproject.toml
  ├─> ADR-Ledger (hardcoded path) ❌ Sem fallback
  └─> Cerebro (código duplicado inline) ❌ Não é dependency
```

**Problemas**:
- `pip install phantom` não instala dependências (quebra)
- Ordem de instalação importa (Neutron antes de Phantom)
- Circular dependency potential (Neutron ← → Phantom)

### Análise de Arquivos Críticos

#### Neutron: SENTINEL (302 LOC)

**Localização**: `/home/kernelcore/arch/neutron/neutron/compliance/sentinel.py`

**Componentes**:
- `ComplianceGuardrail`: Core compliance engine
- `ValidationResult`: Dataclass para resultados
- `AgentOutput`, `EnforcedOutput`: Structured outputs
- `ComplianceViolation`: Violation tracking

**Por que é genérico**:
- Zero lógica específica de Neutron
- Pure functions (text → ValidationResult)
- Configurável via rules (LGPD, SOC2, PCI-DSS)

#### Phantom: CORTEX Chunking (~200 LOC)

**Localização**: `/home/kernelcore/arch/phantom/src/phantom/core/chunker.py`

**Componentes**:
- `SemanticChunker`: Sliding window + boundary detection
- `ChunkMetadata`: Metadata extraction
- Overlap handling, token counting

**Por que é genérico**:
- Semantic chunking é domain-agnostic
- Works on arbitrary text (documents, OSINT data, logs)
- Zero coupling com Phantom-specific logic

#### Phantom: RAG Vector Store (~300 LOC)

**Localização**: `/home/kernelcore/arch/phantom/rag/vectors.py`

**Componentes**:
- `FAISSVectorStore`: FAISS index wrapper
- `EmbeddingGenerator`: Model abstraction
- Hybrid search (semantic + keyword)

**Por que é genérico**:
- FAISS abstractions reutilizáveis
- Embedding models são pluggáveis
- Works com qualquer knowledge domain

### Necessidade de Negócio

Criar **biblioteca compartilhada `nexus-core`** que:
1. Elimina duplicação de código (1,400+ LOC)
2. Torna compliance obrigatório (sem fallback mode)
3. Remove hardcoded paths (portabilidade)
4. Explicita dependências (declarativo via pip)
5. Centraliza testes (>90% coverage na library)

**Objetivo**: Single source of truth para componentes compartilhados, mantendo autonomia dos projetos.

## Decision

Criar biblioteca Python **`nexus-core`** distribuída via pip contendo componentes compartilhados.

## Rationale

### Comparative Analysis

| Critério | Plugin Arch | Microservices | Monorepo | **Shared Library** |
|----------|------------|---------------|----------|-------------------|
| **Latency** | 5-10ms | 20-100ms | <1ms | **<1ms** ✓ |
| **Complexity** | Alta | Muito Alta | Média | **Baixa** ✓ |
| **Autonomia** | Alta | Alta | Baixa | **Alta** ✓ |
| **Maintainability** | Média | Baixa | Alta | **Alta** ✓ |
| **Portabilidade** | Média | Alta | Baixa | **Alta** ✓ |
| **Testability** | Média | Alta | Alta | **Alta** ✓ |
| **Deployment** | Complexo | Muito Complexo | Simples | **Simples** ✓ |
| **Debugging** | Difícil | Muito Difícil | Fácil | **Fácil** ✓ |
| **Overhead** | Médio | Alto | Zero | **Mínimo** ✓ |

**Decision**: Shared Library vence em 8/9 critérios.

### Por que não Microservices?

**SENTINEL como serviço** seria atraente para:
- Escalabilidade horizontal independente
- Language-agnostic (polyglot projects)
- Deploy independente (SENTINEL v2.0 sem rebuild de projetos)

**Mas nosso caso não requer**:
- Volume: ~1,000 compliance checks/day (não precisa de horizontal scaling)
- Latency crítica: Compliance gate deve ser <1ms (in-process call)
- Network = single point of failure (SENTINEL down → all projects blocked)
- Complexity: Service discovery, load balancing, retry logic

**Trade-off**: Sacrificamos escalabilidade teórica (não precisamos) para ganhar simplicidade + latência.

### Por que não Plugin Architecture?

**Plugins** são excelentes para:
- Extensões user-contributed (VS Code extensions, WordPress plugins)
- Hot-swapping (load plugin without restart)
- Version isolation (plugin A v1.0, plugin B v2.0 simultaneously)

**Mas nosso caso não requer**:
- Número de plugins: 4 projetos fixos (não é marketplace)
- Hot-swapping: Deployment em batch (não precisa de runtime loading)
- Isolation: Shared components devem ser consistentes (não queremos SENTINEL v1.0 e v2.0 simultaneamente)

**Trade-off**: Plugins adicionam 5-10ms latency + complexity, ganho zero para nosso cenário.

### Por que não Monorepo?

**Monorepo** resolve:
- Atomic refactoring (rename propagates instantaneamente)
- Dependency graph explícito (Bazel/Pants enforcement)
- Single CI/CD pipeline

**Mas nosso caso valoriza**:
- Autonomia: Phantom evolve independentemente de Neutron
- Build speed: nexus-core rebuild não requer rebuild de todos projetos
- Git workflow: Evitar conflicts em shared files

**Trade-off**: Shared Library dá 90% dos benefícios (single source of truth) sem perda de autonomia.

### Versionamento e Deprecation Strategy

#### Semver Compliance

```
nexus-core 0.1.0  → SENTINEL extraction (Phase 1)
nexus-core 0.2.0  → CORTEX chunking added (Phase 2)
nexus-core 0.3.0  → RAG vector store added (Phase 2)
nexus-core 1.0.0  → Stable release (Phase 3)
```

### Risk Mitigation Details

#### Circular Dependency Prevention

**Rule**: nexus-core NEVER imports from projects.

**Enforcement**:
```python
# CI script: scripts/check_circular_deps.py
import ast
import sys

def check_imports(file_path):
    with open(file_path) as f:
        tree = ast.parse(f.read())

    for node in ast.walk(tree):
        if isinstance(node, ast.Import):
            for alias in node.names:
                if alias.name.startswith(('neutron.', 'phantom.', 'cerebro.', 'spider_nix.')):
                    print(f"ERROR: nexus-core cannot import from projects: {alias.name}")
                    sys.exit(1)

# Run in CI for all nexus_core/*.py files
```
