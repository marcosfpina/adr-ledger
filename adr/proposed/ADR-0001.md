## Context

The current RAG pipeline uses an in-process `CrossEncoderReranker` based on `sentence-transformers`. While functional, this approach is CPU-intensive and blocks the Python GIL, affecting the responsiveness of the main application (Cerebro/Phantom) during high-load reranking of large document sets. Furthermore, the Python implementation lacks the low-level optimizations available in Rust.

A separate project, `cerebro-reranker`, has been developed using Rust (via `pyo3` and `maturin` or direct Rust service) to provide high-performance reranking. We need to integrate this service into the main Cerebro application without introducing tight coupling or strict runtime dependencies that would break the monolithic experience if the service is missing.

## Decision

We will adopt the **Sidecar Pattern** for integrating the reranker service.

1.  **Architecture**: The `cerebro-reranker` service will run as a separate process (sidecar) alongside the main Cerebro application.
2.  **Communication**: Communication will occur via HTTP (REST) over `localhost` on port **8090**.
3.  **Orchestration**: The main `start_cerebro.sh` script will be responsible for detecting, starting, and stopping the sidecar process.
4.  **Fallback Strategy**: The Cerebro application will implement a `CerebroRerankerClient` with a Circuit Breaker pattern. If the sidecar is unreachable or returns an error, the system will transparently fall back to the local in-process `CrossEncoder` implementation.

## Rationale

### Drivers

-   **Performance**: Offloading heavy computation to a Rust-optimized service.
-   **Scalability**: The reranker can be deployed independently on GPU nodes if needed.
-   **Resilience**: The fallback mechanism ensures the system remains functional even if the optimization service fails.

### Alternatives Considered

#### Option A: Direct FFI Binding
-   **Pros:** Lowest latency.
-   **Cons:** Complex build process, potential for segfaults crashing the main app, difficult to manage dependencies (Rust toolchain required for main app build).
-   **Why rejected:** High complexity and coupling.

#### Option B: Serverless / Cloud Function
-   **Pros:** Infinite scale.
-   **Cons:** High latency (cold starts), data privacy concerns (sending data out), cost.
-   **Why rejected:** Latency requirements for interactive RAG.

### Trade-offs

-   **Complexity**: We now manage two processes. This is mitigated by the unified start script.
-   **Deployment**: Requires ensuring `cerebro-reranker` is available or accepting fallback performance.

## Consequences

### Positive

-   Reranking latency expected to drop significantly (target: <50ms for 100 docs).
-   Main application thread remains free for handling other requests.
-   Clear path to GPU acceleration for reranking without modifying the main app.

### Negative

-   Development environment requires pulling two repositories for full functionality.

### Risks

| Risk | Probability | Impact | Mitigation |
|------|-------------|--------|------------|
| Port conflict on 8090 | low | medium | Configurable via env var `CEREBRO_RERANKER_URL` |
| Sidecar startup delay | high | low | Client retries and local fallback |
