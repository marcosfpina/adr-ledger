## Context

O CHAINSCOPE coleta métricas contínuas de ~3000 protocolos DeFi — TVL, preços,
fees, volumes — com frequência horária. Cada ponto é um tuple
`(timestamp, protocol_id, metric_name, value_numeric)`. O padrão de acesso
dominante é por janela temporal.

Era necessário escolher o engine de storage primário antes de implementar
qualquer collector.

## Decision

**Usar TimescaleDB (extensão PostgreSQL) com hypertable `protocol_metrics`,
acessado via pool asyncpg.**

Schema central:

```sql
CREATE TABLE protocol_metrics (
    timestamp       TIMESTAMPTZ NOT NULL,
    protocol_id     TEXT NOT NULL,
    chain_id        TEXT,
    metric_name     TEXT NOT NULL,
    value_numeric   DOUBLE PRECISION,
    value_json      JSONB,
    source_id       TEXT NOT NULL,
    confidence      REAL DEFAULT 1.0,
    PRIMARY KEY (timestamp, protocol_id, metric_name)
);
SELECT create_hypertable('protocol_metrics', 'timestamp',
    chunk_time_interval => INTERVAL '1 day',
    if_not_exists => TRUE);
```

Client: `services/storage/timescale.py` — `TimescaleClient` com pool 2-10 conexões,
`upsert_metrics_batch()` via `executemany`, retry exponencial.

## Rationale

### Drivers

- Dados são time-series → hypertable com chunk pruning reduz I/O em queries temporais
- Volume diário ~260k linhas → adequado para TimescaleDB sem ClickHouse na FASE 1
- asyncpg: client async nativo, ~3x throughput vs psycopg2
- TimescaleDB é drop-in extension: zero risco de adoção

### Alternatives Considered

#### Option A: ClickHouse puro
- **Pros:** Throughput analítico superior, colunar nativo
- **Cons:** Sem ACID, complexidade operacional maior
- **Why rejected:** ClickHouse existe no stack para analytics (FASE 3); duplicaria storage sem ganho

#### Option B: InfluxDB
- **Pros:** Nativo time-series, InfluxQL familiar
- **Cons:** Licença proprietária v3, sem SQL padrão, ecossistema Python menor
- **Why rejected:** Vendor lock-in, ausência de SQL

#### Option C: PostgreSQL vanilla
- **Pros:** Sem extensão extra
- **Cons:** Particionamento manual, sem continuous aggregates
- **Why rejected:** TimescaleDB drop-in — ganho real com custo zero

### Trade-offs

- Dependência de versão do TimescaleDB → pinado no docker-compose

## Consequences

### Positive

- Chunk pruning automático reduz I/O em queries temporais
- Compression após 7 dias (~10x espaço)
- Retention 2 anos declarativa
- ON CONFLICT DO UPDATE elimina duplicatas de re-coletas
- Retry exponencial em TooManyConnectionsError

### Negative

- Extensão TimescaleDB requer setup na instância PostgreSQL

### Risks

| Risk | Probability | Impact | Mitigation |
|------|-------------|--------|------------|
| TimescaleDB version mismatch | low | medium | Versão pinada no docker-compose |
| Pool exhaustion sob carga | low | medium | Retry com backoff implementado |
