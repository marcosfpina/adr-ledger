## Context

A stack inteligente (SPECTRE, PHANTOM, CEREBRO, NEUTRON) precisa de uma base de conhecimento estruturada que seja:

1. **Parseável** - Agentes precisam consumir decisões programaticamente
2. **Imutável** - Audit trail para compliance e debugging
3. **Soberana** - Sem dependência de ferramentas SaaS
4. **Human-readable** - Devs e stakeholders precisam entender
5. **Versionada** - Evolução de decisões ao longo do tempo

O problema central: conhecimento arquitetural estava fragmentado em múltiplas ferramentas (Notion, Obsidian, cabeça do desenvolvedor), impossibilitando tanto auditoria quanto consumo automatizado por sistemas inteligentes.

## Decision

Adotamos **Git como sistema operacional** para decisões arquiteturais:

- **Formato**: YAML frontmatter + Markdown body (parseável + legível)
- **Storage**: Git repository com estrutura `adr/{status}/ADR-XXXX.md`
- **Schema**: JSON Schema para validação e tipagem
- **Tooling**: CLI bash + Parser Python para operações
- **Output**: JSON knowledge base para consumo por agentes

A transformação de ADRs em "knowledge law" acontece via AST parsing:
```
ADR.md → Parser → ADRNode → KnowledgeFragment → CEREBRO/SPECTRE/PHANTOM
```

## Rationale

### Por que Git?

| Propriedade | Benefício |
|-------------|-----------|
| **Imutável** | Cada commit é um registro permanente |
| **Queryable** | `git log`, `grep`, shell tools |
| **Distribuído** | Funciona offline, qualquer host |
| **Atômico** | Mudanças relacionadas em um commit |
| **Auditável** | `git blame` mostra quem decidiu o quê |
| **Portável** | Move de GitHub para Gitea em minutos |
| **Assinável** | GPG-sign prova autenticidade |

### Knowledge Sovereignty

> "Not your keys, not your crypto" → "Not your repo, not your architectural rationale"

Empresas documentam decisões no Confluence (Atlassian), Notion (terceiros), Google Docs (Google). O conhecimento fundacional está em silos proprietários.

Git + ADRs estruturados = **sovereign knowledge base**. A organização é dona do seu próprio raciocínio, cryptographically verified, distribuído entre stakeholders.

## Consequences

### Impacto nos Sistemas Inteligentes

- **CEREBRO**: Ingere `knowledge_base.json` para RAG queries
- **SPECTRE**: Analisa `spectre_corpus.json` para sentiment/patterns em decisões
- **PHANTOM**: Usa `phantom_training.json` para classificação de novas ADRs
- **NEUTRON**: Enforça compliance baseado em ADRs aceitas

### Workflow Típico

```bash
# 1. Criar nova decisão
adr new -t "Use K3s for Kubernetes" -p NEUTRON -c major

# 2. Editar e preencher
$EDITOR adr/proposed/ADR-0006.md

# 3. Review e aprovação
git commit -S -m "ADR-0006: Proposta de K3s"

# 4. Aceitar após review
adr accept ADR-0006

# 5. Sincronizar knowledge
adr sync  # → Atualiza knowledge_base.json

# 6. CEREBRO agora conhece a decisão
cerebro query "por que usamos K3s?"
```
