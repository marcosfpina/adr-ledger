## Context

SPECTRE requires Kubernetes deployment for production with multiple environments (dev, prod). Initial exploration led to creating a full Helm chart (17 files, 813 lines) with comprehensive features. However, during implementation, we identified that Nix provides superior guarantees for reproducible deployments and eliminates external dependencies like Docker daemon.

The project already uses Nix flakes for development environment, making a Nix-first approach for infrastructure a natural fit. The key question became: should we maintain Helm charts or migrate to pure Nix orchestration?

## Decision

**We adopt Nix as the primary Kubernetes orchestration mechanism**, replacing Helm charts with declarative Nix modules that generate Kubernetes manifests. This includes:

- `nix/lib/k8s.nix`: Helper functions for manifest generation
- `nix/kubernetes/*.nix`: Resource definitions (Deployment, Service, ConfigMap, Ingress)
- `nix/images/spectre-proxy.nix`: Container image builder using dockerTools
- Flake integration with packages (manifests) and apps (deploy scripts)
- Isolated operations via `nix develop .#kubernetes`

Helm chart remains available for teams preferring traditional tooling, but Nix is the recommended and maintained approach.

## Rationale

### Drivers

1. **Reproducibility**: Nix guarantees bit-for-bit reproducible builds via content-addressable store
2. **Declarative Config**: Everything is code - no imperative kubectl commands
3. **No Docker Daemon**: Build OCI images without Docker using dockerTools.buildLayeredImage
4. **Type Safety**: Nix language catches configuration errors at build-time
5. **Environment Parity**: dev/prod configs share same codebase, only parameters differ
6. **Hermetic Builds**: All dependencies pinned in flake.lock, no version drift

### Alternatives Considered

#### Option A: Helm Charts
- **Pros:**
  - Industry standard, large ecosystem
  - Rich templating with values.yaml
  - helm diff, helm rollback built-in
  - Wide team familiarity
- **Cons:**
  - Requires Tiller (Helm 2) or helm client (Helm 3)
  - Templates can become complex (Go templates)
  - No build-time validation of generated YAML
  - Version drift between Chart.yaml and actual images
- **Why rejected:** Lacks reproducibility guarantees, requires external tooling, imperative workflow

#### Option B: Kustomize
- **Pros:**
  - Native kubectl integration
  - Overlay-based composition
  - No templating, pure YAML
- **Cons:**
  - Limited validation
  - Verbose overlay duplication
  - No container image building
- **Why rejected:** Doesn't solve reproducibility, still requires Docker daemon

#### Option C: Plain YAML + CI/CD
- **Pros:**
  - Simple, no abstraction
  - Direct kubectl apply
- **Cons:**
  - High duplication between environments
  - No parameterization
  - Manual version management
- **Why rejected:** Not scalable, error-prone

### Trade-offs

- **Accept**: Steeper learning curve for Nix language vs Helm templates
- **Accept**: Smaller community and fewer examples than Helm ecosystem
- **Accept**: JSON output instead of YAML (Kubernetes accepts both equally)
- **Gain**: Build-time correctness verification
- **Gain**: Hermetic builds without network dependencies during deployment

## Consequences

### Positive

1. **Zero Configuration Drift**: Nix store guarantees same deployment everywhere
2. **Fast Iteration**: `nix build .#kubernetes-manifests-dev` validates instantly
3. **Atomic Rollbacks**: `nix run .#deploy-dev` uses flake.lock pinned versions
4. **No Docker Daemon**: CI/CD doesn't need privileged containers for image building
5. **Developer Isolation**: `nix develop .#kubernetes` provides kubectl, helm, k9s without polluting host
6. **Pre-configured Environments**: dev/prod configs in `nix/kubernetes/default.nix` with sensible defaults
7. **Composable**: Can mix Nix-generated manifests with existing Helm charts if needed

### Negative

1. **Learning Curve**: Team must learn Nix language syntax and evaluation model
2. **Tooling Gap**: No `nix diff` equivalent to `helm diff`, must use kubectl diff manually
3. **Debugging**: Nix evaluation errors can be cryptic compared to YAML syntax errors
4. **Community Size**: Fewer K8s+Nix examples than Helm chart repositories

### Risks

| Risk | Probability | Impact | Mitigation |
|------|-------------|--------|------------|
| Team resistance to Nix | medium | medium | Provide training, maintain Helm fallback |
| Nix evaluation bugs | low | high | Extensive testing, gradual rollout |
| Missing Nix expertise during incidents | medium | high | Document patterns, cross-train team |
| Ecosystem tool incompatibility | low | low | Use kubectl directly, avoid Nix-specific workflows in prod |
